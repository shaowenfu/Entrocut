# 08. Feature Development Principles（功能开发原则）

本项目不再维护按阶段推进的固定开发路径，仅保留 `Feature-driven Development（按功能切片开发）` 原则。

## 1. 基本原则

1. 一个功能切片只解决一类用户能力，不跨切片并发扩张。
2. 先定义 `Contract（契约）`，再实现 `core/server/client`。
3. 每个切片都必须跑通最小 `E2E（端到端）` 链路。
4. 不为未来做预实现，不引入当前无调用方的复杂度。

## 2. 切片执行顺序（固定）

1. `Scenario（场景）`：明确用户动作、系统反馈、验收标准。
2. `Function（功能）`：拆成可交付的最小能力单元。
3. `Contract`：定义输入输出、字段语义、错误码。
4. `Implementation`：先 `core/server`，后 `client` 替换 `mock`。
5. `E2E`：验证从界面触发到结果反馈的完整闭环。

## 3. 每个切片的 DoD（完成定义）

1. 文档更新完成：`Contract/API/错误语义` 可被评审。
2. 三端类型一致：`TypeScript + Pydantic` 定义无漂移。
3. 目标路径无 `mock`：该功能不再依赖假数据。
4. 至少一条可重复执行的 `E2E` 用例通过。
5. 日志可定位：关键请求具备 `request_id` 追踪。

## 4. 禁止事项

1. 禁止跳过契约直接堆实现。
2. 禁止在一个切片内同时重构 UI 架构和业务逻辑。
3. 禁止用非结构化文本承载决策结果（必须结构化返回）。
4. 禁止以临时分支逻辑替代错误语义设计。

## 5. 推荐切片粒度

1. `Health & Session`：服务健康状态和会话初始化。
2. `Chat Decision Loop`：对话请求与决策回包。
3. `Render Preview`：契约驱动预览渲染。
4. `Ingest & Index`：导入、切片、索引入库。
5. `Refine Patch`：局部替换和微调。
6. `Export Lock`：导出与编辑锁。

以上仅为示例，不是固定路线图。实际开发以“当前要交付的功能闭环”为唯一拆分依据。
