# 从清场式重构到最小闭环跑通：我的开发日记

## 写在前面

今天我决定把这段时间的项目推进过程完整记下来。不是为了做一份对外汇报，而是为了给未来的自己留下一条清晰的轨迹：我为什么要做这么激进的删除，删除之后到底保住了什么，又是如何从一片废墟里，把项目重新搭成一个可以继续往前走的系统。

这份记录采用第一人称，因为这本来就是一次强烈带有主观判断的工程决策。我需要对自己的判断负责，也需要把这些判断背后的动机、做法和反思讲清楚。

---

## 一、我为什么选择“彻底删除重构”

在这轮重构开始之前，项目已经出现了一个很危险的状态：表面上功能越来越多，实际上核心设计越来越模糊。

我当时最强烈的感受不是“功能少”，而是“错误逻辑在不断堆叠”。

具体表现有几个：

1. 旧的 `core/server` 业务实现已经偏离了真正想做的产品路径。
2. `mock`、旧测试脚本、旧 `client contract/api/events` 层之间互相牵扯，维护成本越来越高。
3. 很多代码不是在表达真实边界，而是在补旧设计留下来的洞。
4. 我已经越来越难回答一个基本问题：这个系统最小可工作的主链路到底是什么。

这时候继续“小修小补”其实是最危险的，因为它会让我误以为自己在推进，实际上是在给错误结构续命。

所以我做了一个激进但必要的决定：先清场，再重建。

---

## 二、清场之后，我到底保留了什么

这次删除不是情绪化地“全部推倒重来”，而是一次边界重设。

我保留下来的主干非常明确：

1. 前端原型 `UI`
2. 本地 `prototype mode` 状态层
   - `useLaunchpadStore.ts`
   - `useWorkspaceStore.ts`
   - `prototypeWorkspace.ts`
3. 最小 `FastAPI skeleton`
   - `core/server.py`
   - `server/main.py`
4. 当前阶段说明和最小启动脚本
   - `core/README.md`
   - `server/README.md`
   - `dev_up.sh`
   - `smoke_test.sh`

删除掉的内容同样非常明确：

1. `core/app`、`server/app` 全部旧业务目录
2. 旧测试
3. 旧 `client` 中的 `contract/api/events` 层
4. 旧 `phase45` 和单元测试脚本

我保留这些，是因为它们代表了当前最有价值的三类资产：

1. 用户界面的交互原型
2. 前端已经隐含出来的产品状态机
3. 后端的最小运行骨架

换句话说，我不是在保留“代码量”，我是在保留“最接近真实产品边界的东西”。

---

## 三、清场后的第一个判断：先不要写后端能力，先定义状态和契约

清场之后，我没有马上冲去补 `Core` 能力，也没有急着接真实算法链路。

我先问了自己一个问题：

如果现在重新开始，我最先该固定的是什么？

最后我的答案是：

1. 先固定前端状态机
2. 再固定 `Core API/WS contract`
3. 最后才是实现

这是一个非常关键的转向。

因为我很快意识到，当前仓库里真正已经“长出来”的，并不是后端业务逻辑，而是前端状态模型。

`Launchpad` 和 `Workspace` 已经通过状态字段表达出了第一版产品流程：

1. 项目创建
2. 素材导入
3. `chat`
4. 分镜生成
5. 导出
6. 错误恢复

如果我不先把这套状态关系说清楚，后面无论写 `API`、写事件流，还是接真实算法，都会再次偏航。

所以这轮工作的真正起点，不是“实现什么功能”，而是“重新定义系统边界”。

---

## 四、我重新理解了“状态管理”这件事

在这个过程中，我有一个很重要的认识变化。

以前很容易把状态管理理解成：

“是不是把很多 `if` 集中到一个地方维护就行了？”

但这次我越来越清楚，状态管理的本质根本不是集中管理 `if`，而是：

1. 定义系统当前处于什么状态
2. 定义什么事件会驱动状态变化
3. 定义哪些状态组合是合法的，哪些是非法的
4. 让 `UI` 从状态派生，而不是从零散布尔值拼凑

这个认识非常重要，因为它直接影响后面的设计方式。

我不再从“页面上哪些东西在变”出发，而是从下面几个问题倒推状态：

1. 系统里有哪些对象会随时间变化
2. 哪些过程有明确的生命周期
3. 哪些动作会受到约束
4. 哪些失败需要被恢复

这套方法最后落到了两份非常关键的文档里：

1. `docs/launchpad_state_model.md`
2. `docs/workspace_state_model.md`

这两份文档帮我把原型里的隐式逻辑，整理成了显式的状态模型和状态切换规则。

---

## 五、Launchpad：我先把入口流程做成一个干净的状态机

`Launchpad` 是整个系统的入口，所以我先从这里下手。

我没有直接改代码，而是先把它拆成几条关键流程：

1. 加载最近项目
2. 检查本地 `Core` 是否可用
3. 创建空项目
4. 上传素材并创建项目
5. 进入 `Workspace`

接着我把状态收敛成几个非常明确的枚举：

1. `projectsLoadState`
2. `systemStatus`
3. `createState`
4. `importState`
5. `navigationState`

然后我要求自己遵守一个原则：

不要再用一堆散乱的布尔值直接驱动页面，而是由这些主状态派生出：

1. 是否可点击
2. 是否显示 loading
3. 是否允许继续创建
4. 是否应该显示错误

这一步的好处非常直接：

我终于可以一眼看出 `Launchpad` 现在处于什么阶段，而不是靠推测多个变量组合。

后面我把这套状态模型落实到 `useLaunchpadStore.ts`，并收敛成 `event -> reducer` 的事件驱动写法。这样一来，页面不再到处散落 `set()`，而是通过统一的状态更新入口推进流程。

---

## 六、Workspace：我把“业务流程状态、连接状态、后台任务状态”拆开了

如果说 `Launchpad` 解决的是入口问题，那么 `Workspace` 解决的就是整个产品核心流程。

这里我最先看到的风险，是几个不同层次的状态混在了一起：

1. 业务流程状态
2. `WebSocket` 连接状态
3. 后台任务状态
4. 页面兼容布尔状态

如果不把这几层拆开，后面一接真实后端，前端一定会乱。

所以我最终把 `Workspace` 理解成三层状态：

1. `workflowState`
   - 管业务阶段，比如 `awaiting_media`、`chat_thinking`、`ready`、`rendering`
2. `eventStreamState`
   - 管事件流连接，比如 `disconnected`、`connecting`、`connected`
3. `activeTask`
   - 管当前后台任务，比如 `ingest`、`chat`、`render`

同时，我又补了：

1. `loadState`
2. `chatState`
3. `lastError`

这套模型最后也被写成了事件驱动结构，并落实到 `useWorkspaceStore.ts`。

我保留了一些兼容字段，但把它们降级成“派生层”，而不是“真实状态来源”。然后又进一步把 `WorkspacePage.tsx` 改成直接消费：

1. `loadState`
2. `chatState`
3. `activeTask`
4. `workflowState`

这一步很重要，因为它意味着页面终于和状态模型对齐了。

---

## 七、我为什么要先写 Core API/WS contract

当 `Launchpad` 和 `Workspace` 的状态模型稳定下来以后，我开始写 `docs/contracts/01_core_api_ws_contract.md`。

这一步我非常确定不能省略。

原因很简单：

如果前端和后端之间没有稳定契约，那么：

1. 前端会按自己的想象写状态
2. 后端会按自己的想象写接口
3. 接起来的时候才发现语义对不上
4. 最后又会陷入一边改前端、一边改后端的返工泥潭

所以我先文档化了几件事：

1. 最小资源模型
   - `Project`
   - `Asset`
   - `Clip`
   - `StoryboardScene`
   - `ChatTurn`
   - `Task`
   - `WorkspaceSnapshot`
2. 最小 `HTTP API`
3. 最小 `WebSocket event stream`
4. 错误语义和前端状态映射

我把这份契约当成边界，而不是实现说明。

也就是说，我要求自己用一种心态来写它：

哪怕未来完全重写实现，这份契约也应该尽量不变。

---

## 八、Core in-memory：我没有急着上真实能力，而是先做假实现闭环

契约写完之后，我没有立刻接：

1. 真实 `FFmpeg`
2. 真实 `Embedding`
3. 真实 `DashVector`
4. 更复杂的 `server auth/proxy`

我选择先扩 `core/server.py`，做一个单文件 `in-memory` 假实现。

这不是偷懒，而是明确地在追求最短反馈路径。

我要的不是一开始就“能力真实”，我要的是先验证：

1. 前端状态机是不是合理
2. 契约能不能驱动最小闭环
3. 事件流的语义是否稳定

所以这版 `Core` 的重点不是业务深度，而是流程完整性。

我把这些接口先跑起来：

1. `GET /health`
2. `GET /api/v1/runtime/capabilities`
3. `GET /api/v1/projects`
4. `POST /api/v1/projects`
5. `GET /api/v1/projects/{project_id}`
6. `POST /api/v1/projects/{project_id}/assets:import`
7. `POST /api/v1/projects/{project_id}/chat`
8. `POST /api/v1/projects/{project_id}/export`
9. `WS /api/v1/projects/{project_id}/events`

同时我在里面补了：

1. `request_id middleware`
2. 统一错误结构
3. `in-memory store`
4. 假任务推进
5. 快照与事件推送逻辑

这版后端最让我满意的地方不是“功能丰富”，而是它开始像一个真正的边界服务了。

---

## 九、从 prototype 切到真实 Core：这一步比我想象中更关键

后端契约和假实现起来之后，我开始把前端从 `prototypeWorkspace` 接到真实 `Core`。

这里我做了几件核心事情：

1. 新增 `coreClient.ts`
2. 让 `Launchpad` 通过真实 `GET /projects` 拉最近项目
3. 让 `Launchpad` 通过真实 `POST /projects` 创建项目
4. 让 `Workspace` 通过真实 `GET /projects/{id}` 初始化
5. 让 `Workspace` 通过真实 `WS /events` 接收快照和增量事件

在这个过程中，我很明显地感受到一件事：

只有当原型真的接到真实边界上，很多隐藏问题才会暴露出来。

比如：

1. 文件选择器的取消/成功竞态
2. `WebSocket` 初始化在开发环境下被重复打开又关闭
3. 旧页面逻辑对兼容字段的依赖
4. 不同来源的状态语义不一致

这一步把“看起来能工作”和“真的能联起来工作”区分开了。

---

## 十、联调让我暴露出两个层次的问题：主链路问题与体验问题

在用户配合下，我做了多轮联调。

一开始最严重的问题是：

1. 浏览器选中文件后没有真正创建项目
2. `WebSocket` 事件流不稳定

这些都属于主链路问题，必须先解决。

### 1. 文件选择器竞态

根因是文件选择框关闭后，`focus` 事件先于 `change` 触发，旧实现把已选文件误判为取消。

我修复后，浏览器上传视频创建项目终于打通。

### 2. WebSocket 初始化竞态

`React StrictMode` 下会触发双初始化，导致同一 `project_id` 的连接被重复创建，然后旧流程又把它提前关掉。

我做了两件修复：

1. 同一项目如果已有 `CONNECTING/OPEN` 连接，就不再重复建
2. 给初始化流程增加 `load token`，忽略过期初始化结果

修完之后，`chat`、`assets import`、`export` 的事件链全部恢复正常。

然后联调进入了第二层问题，也就是体验问题：

1. 没有缩略图
2. 点击素材/片段/分镜，预览区不联动
3. 播放器只是程序跳转器，不是真正可交互播放器

这些不再是“能不能跑”的问题，而是“用户能不能感知和验证系统行为”的问题。

---

## 十一、我决定优先补 Preview，而不是继续做别的 UI 修补

当最小主链路跑通以后，我没有继续去补零碎的页面细节，而是优先做了 `Preview` 联动。

我的判断依据很直接：

如果用户生成了 `Storyboard`，却无法点击某个 `Asset / Clip / Scene` 去看对应内容，那么整个工作台虽然“状态上可用”，实际上仍然缺少最关键的感知闭环。

所以我优先补了：

1. 本地媒体源注册
2. 预览播放器联动
3. 缩略图支持
4. 播放控制与 `seek`

这里我没有先做复杂的持久化媒体层，而是用了一个非常务实的做法：

通过 `localMediaRegistry.ts` 在当前前端会话内登记本地文件源，并基于这些源生成预览地址和缩略图。

这样做的优点是：

1. 复杂度低
2. 路径短
3. 足以支撑当前联调

代价也很明确：

1. 缩略图不是持久资产
2. 媒体源依赖当前会话
3. `Storyboard -> Clip` 映射暂时还是近似映射

但这完全符合当前阶段目标。

---

## 十二、到目前为止，我实际跑通了什么

经过这一轮重构和联调，我最终确认跑通了两个关键闭环。

### 闭环一：系统流程闭环

1. `Launchpad -> 创建项目`
2. `Workspace -> 初始化快照`
3. `chat -> 生成 assistant decision + storyboard`
4. `assets import -> 补传素材`
5. `export -> 返回导出结果`
6. 全程由 `WebSocket event stream` 驱动状态推进

### 闭环二：内容感知闭环

1. 点击 `Asset` 播放素材
2. 点击 `Clip` 播放对应片段
3. 点击 `Storyboard` 卡片跳转预览
4. 播放/暂停状态同步
5. 进度条支持点击和拖动
6. `Assets / Clips / Storyboard` 都有缩略图

这意味着项目已经从“文档 + 原型 `UI` + 空骨架”，变成了一个：

能够创建、导入、生成、导出、预览、联动的最小工作系统。

虽然它还是假实现居多，但关键是：

它已经有了正确的骨架。

---

## 十三、这轮推进中我最看重的几个方法论

回头看，这次推进最有效的不是某个单点技术，而是几个方法论上的坚持。

### 1. 先删到边界清楚，再重建

错误结构积累到一定程度以后，继续修补只会提高沉没成本。真正有效的方式是先把错误边界切掉。

### 2. 先定义状态，再定义契约，最后写实现

如果顺序反过来，实现一定会反向污染边界。

### 3. 不抢跑真实能力

先做 `in-memory` 假实现闭环，比急着上真实算法链路更有价值，因为它能更早暴露边界问题。

### 4. 主链路优先于体验，体验优先于精致

我先解决“能不能跑”，再解决“能不能感知”，最后才考虑“好不好看、真不真实”。

### 5. 用文档收敛思路，而不是只靠口头理解

这次几份状态文档和契约文档的价值非常高，它们让我避免了反复在脑中重算系统边界。

---

## 十四、我也看到了当前结构的明确限制

虽然这轮结果是正向的，但我不想把当前系统看得过于乐观。

它现在仍然有一些明确限制：

1. `Core` 还是 `in-memory`
   - 没有持久化
   - 重启即丢
2. `StoryboardScene -> Clip` 还没有显式映射字段
   - 当前只是按索引近似映射
3. 缩略图和媒体源仍然是会话内能力
   - 不是后端稳定资产
4. `Storyboard` 仍然是 `read-only`
   - 还没有结构化编辑入口
5. `server` 依旧只是骨架
   - 认证、代理、真实能力编排都还没接

这些问题不是意外，而是当前阶段刻意接受的非目标。

我现在反而更在意的是：

这些边界有没有被说清楚。

只要边界清楚，后面的推进就有方向；最怕的是边界不清，却假装系统已经完整。

---

## 十五、如果我总结这轮工作的核心收获

如果要我用一句话总结这轮重构，我会这样说：

我不是把项目“修好了”，而是把项目重新放回了一个可以持续推进的轨道上。

这条轨道的核心不是某个具体实现，而是三件事：

1. 清晰的状态模型
2. 稳定的本地契约
3. 已被联调验证的最小闭环

这三件事一旦成立，后面的工作就不再是“盲修”，而是沿着已经建立的边界，逐步替换掉假实现、补齐真实能力、扩展编辑能力。

对我来说，这比一开始堆很多“看起来更厉害”的功能更重要。

---

## 十六、我对下一阶段的判断

现在如果继续推进，我认为最合理的方向不是继续做零散 `UI polish`，也不是立刻接复杂后端能力，而是进入“结构化编辑能力”的阶段。

我当前最想做的几件事是：

1. `Storyboard selection/edit intent`
   - 让用户可以明确选中某个 `scene` 或 `clip`
   - 让 `chat` 不再只支持全局指令，还能支持局部编辑意图
2. `Scene -> Clip` 显式映射
   - 把现在的近似预览映射升级成稳定契约
3. 媒体与缩略图的稳定化
   - 决定哪些字段应该由 `Core` 提供，哪些继续由前端会话层承担

如果这一步走稳，项目就会从“能演示一个流程”，进入“能承载真实编辑行为”的阶段。

---

## 最后的话

我写这份开发日记，不是为了把过程美化成一条线性成功故事。

事实恰恰相反，这轮推进之所以重要，就是因为它经历了一个先承认错误结构、再主动清场、再逐步重建边界的过程。

如果未来我再回来看这段记录，我希望自己记住的不是某个函数名或某个提交，而是这件事：

当系统开始偏离核心设计时，最重要的不是更努力地往上堆代码，而是先停下来，重新回答：

1. 目标是什么
2. 边界是什么
3. 最小闭环是什么
4. 当前哪些东西值得保留，哪些东西必须果断舍弃

这次我做对的地方，不是“写得快”，而是终于愿意对这些问题给出明确答案。
