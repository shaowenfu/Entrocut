# 2026-03-30 本地数据层落地日志

今天我把 `EntroCut` 的本地数据层，真正从“方向明确”推进到了“骨架落地”。

这件事我之前已经在文档里讲清楚很多次了：

1. 本地权威事实要放在 `core`
2. 结构化业务数据要进 `SQLite`
3. 原始素材默认只存路径引用
4. 项目中间文件要进入项目工作目录
5. 敏感凭证不能长期留在前端 `localStorage`

但今天之前，这些更多还停留在架构判断和设计文档里。

今天我做的事情，是把它们一条一条落成代码。

---

## 一、我先把第一阶段做成了真正的本地持久化骨架

我没有一开始就去重写整个 `core`。

我很清楚，如果这一步贪大，最后一定会把 `server.py` 打散得很难读，而且还会把现有主链搞崩。

所以我刻意控制了第一阶段的范围，只做四件事：

1. 抽出 `storage_paths`
2. 抽出 `workspace_manager`
3. 抽出最小 `SQLite repository`
4. 在 `create_project` 时自动初始化项目工作目录

最后形成的最小骨架是：

1. [core/storage_paths.py](/home/sherwen/MyProjects/Entrocut/core/storage_paths.py)
2. [core/workspace_manager.py](/home/sherwen/MyProjects/Entrocut/core/workspace_manager.py)
3. [core/local_state_repository.py](/home/sherwen/MyProjects/Entrocut/core/local_state_repository.py)

我对这一步的要求很简单：

`不改变现有 API 语义，但让 core 不再是纯内存态。`

这一步做完以后，`create_project` 已经不只是往 Python 字典里塞一条记录了。

它会同时：

1. 创建本地数据库
2. 创建项目工作目录
3. 初始化 `thumbs / preview / exports / temp / proxies`
4. 把项目初始状态同步写入本地库

对我来说，这就是本地数据层真正开始落地的标志。

---

## 二、第二阶段我把“整条 record 快照持久化”升级成了正式表驱动

第一阶段能跑，但我很清楚那还只是一个过渡骨架。

如果继续让整条 `record_json` 当主链事实源，后面一定会越来越混乱。

所以第二阶段我做的核心判断是：

`必须尽快让 SQLite 里的正式表成为主链，而不是继续依赖一整块快照。`

我最后收口出来的最小表是：

1. `projects`
2. `edit_drafts`
3. `chat_turns`
4. `tasks`
5. `project_runtime`
6. `assets`

这里我没有为了“数据库优雅”去过度范式化。

我还是坚持了之前定下来的原则：

1. `EditDraft` 先用 `draft_json`
2. `assistant turn / task payload / export result` 允许 JSON
3. 先解决恢复性和一致性，再慢慢细拆

这一步对我来说非常重要，因为它意味着：

`core` 的本地数据层开始真正像一个本地后端，而不是一个带 SQLite 影子的内存服务。`

---

## 三、第三阶段我把工作目录和文件落点真正收口了

这一块其实是我今天感受最明确的地方。

因为在这之前，文档里一直说：

1. 原始素材默认只存引用
2. 导出和中间产物应该进入项目工作目录

但如果代码里还继续往 `/tmp/...` 写文件，这件事就根本没落地。

所以第三阶段我只抓三件事：

1. 统一项目子目录接口
2. 让 `export` 不再落 `/tmp`
3. 用测试把“素材只存引用、不复制”锁住

最后的结果是：

1. 导出产物已经写进 `projects/<project_id>/exports/`
2. 项目工作目录真正开始承接产物
3. 原始素材仍然保留 `source_path`
4. 工作目录里不会偷偷复制原始素材文件

我很满意这一步，因为它让“本地工作目录”从概念变成了运行时真实存在的东西。

这一步之后，文档里那句：

`reference-only`

终于不只是设计偏好，而是代码层面的行为约束了。

---

## 四、第四阶段我把凭证迁移做到了 Electron 原生层

这一块我一开始就不想用新的第三方依赖把问题变复杂。

我优先看的不是“有没有现成库”，而是：

`当前 Electron 主进程已经有 IPC 和 preload 桥，那我能不能用系统自带能力先做最小闭环。`

最后我选的是：

1. 在 Electron 主进程里加最小安全凭证仓库
2. 用 `safeStorage` 加密本地凭证文件
3. 通过 `preload + electronBridge` 暴露 `get / set / delete`

然后在前端认证链里，我把逻辑收成了三步：

1. 启动时优先读安全存储
2. 如果只在 `localStorage` 里发现旧 token，就做一次性迁移
3. 以后新的 token 只写安全存储

这一块我刻意没有改动 `client -> core /api/v1/auth/session` 这条同步链。

原因很简单：

`我想改的是本地凭证保存方式，不是 server/core 的鉴权协议。`

这样风险最小，也更符合第四阶段“只替换存储，不重写协议”的原则。

---

## 五、我把 core 的 auth session 也接回了本地持久化层

做到第四阶段之后，我又顺手把另一个不该继续停在内存里的点补掉了：

`CoreAuthSessionStore`

之前它还是纯进程内内存。

这意味着：

1. `client` 已经把 token 安全保存了
2. 但 `core` 一旦重启，又会失去本地 access session

这个状态很不对称。

所以我最后又补了一张最小表：

1. `core_auth_session`

这里只保存：

1. `access_token`
2. `user_id`
3. `updated_at`

我没有让 `core` 持有 `refresh_token`，因为这个边界之前已经在 auth 设计里讲得很清楚了：

`refresh ownership（刷新责任）` 还是在 `client`

`core` 只消费同步过来的最新 `access token`

这一步做完以后，本地数据层的职责就更一致了：

1. `client` 负责安全持有完整桌面登录态
2. `core` 负责持有调用本地后端主链所需的会话镜像

这个分工我觉得是干净的。

---

## 六、最后我把 project_records 这个过渡表清掉了

我很清楚，这个过渡表如果不及时清理，后面一定会变成技术债。

所以在整个本地数据层都接稳之后，我没有再继续“保守兼容”，而是做了最后一步收尾：

1. 如果检测到旧的 `project_records`
2. 就在初始化时一次性迁移
3. 然后直接删除表

这一步的意义不在于多写了一点 SQL，而在于：

`我不想让一个过渡期产物长期留在主线里制造歧义。`

到这一步，本地数据层的主链终于清晰了：

1. 结构化事实走正式表
2. 工作目录承接产物
3. 凭证走安全存储
4. `core` 只保留 access session 镜像

---

## 七、这次我最满意的不是“功能变多了”，而是边界终于变清楚了

回头看今天这一整轮工作，我最满意的地方其实不是新增了几张表，也不是多了几个模块。

我最满意的是，几个之前经常混在一起的问题，现在终于被拆开了：

1. 什么是本地权威事实
2. 什么是项目产物
3. 什么是原始素材引用
4. 什么是桌面安全凭证
5. 什么是 `client` 的职责
6. 什么是 `core` 的职责

这次之后，我对整个本地数据层的判断比之前更稳了：

`EntroCut` 现在已经不再只是“有一个本地 FastAPI 服务”，而是开始拥有一个真正的桌面本地后端。`

---

## 八、给未来的我留几句最重要的话

1. 不要再把 `client` 当成本地事实源
2. 不要让 `core` 回到纯内存态
3. 不要再把导出和中间产物写回 `/tmp`
4. 不要把 `refresh token` 下沉到 `core`
5. 不要再引入新的过渡层而不及时清理

如果下一步继续推进，我现在最明确的判断是：

`本地数据层主线已经基本站住，接下来最值得投入的还是 core/chat 的生产级执行闭环，而不是再扩一堆新的存储花样。`
