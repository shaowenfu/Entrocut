# 2026-03-31 项目状态管理收口日志

今天这轮工作的目标很明确：

`把 EntroCut 从“单一 workflow_state 驱动”的残余兼容态，真正收口到“按事实分层驱动”的公开契约。`

这不是一次简单的字段改名。

如果只是把 `workflow_state` 换成另一组枚举名，问题不会消失，因为真正的问题从来都不是“名字不够好”，而是：

1. 把素材处理状态、`agent runtime（智能体运行时）` 状态、导出状态、错误状态压进了同一个轴
2. 前端和后端都被迫对这个单轴状态做过度推理
3. 空项目、部分素材处理中、聊天失败但草案仍可继续编辑，这些真实场景都表达不干净

所以我今天做的事情，本质上是在把“状态是什么”重新拉回到第一性原理。

---

## 一、我先把公开契约彻底改成事实分层

这一步我最在意的是：

`公开契约只能暴露对前端真正有用、而且不绑定实现细节的事实。`

最后收口出来的权威状态层是：

1. `summary_state`
2. `media_summary`
3. `runtime_state`
4. `capabilities`
5. `active_tasks`

这里面每一层只回答一种问题：

1. `summary_state`
   - 当前整体阶段是什么
2. `media_summary`
   - 素材是否完成切分、向量化、可检索
3. `runtime_state`
   - `agent` 当前在想什么、看什么、卡在哪里
4. `capabilities`
   - 当前允许做什么
5. `active_tasks`
   - 现在到底有哪些异步过程在跑

这一步之后，`Project` 不再承担那个历史遗留的“大总状态”职责。

对我来说，这才是一个健康的 `API contract（接口契约）`。

---

## 二、后端真正完成了“公开去 workflow_state，内部保兼容”

这一轮里我做了一个很关键的取舍：

`不要再让 workflow_state 参与公开读写，但也不要为了“概念纯洁”立刻把 SQLite 老列炸掉。`

所以我最后的方案是：

1. `schemas.py` 里把 `ProjectModel` 改成只暴露 `summary_state`
2. `store.py` 的 `workspace_snapshot`、事件推送、错误载荷全部围绕新状态层工作
3. `state.py` 里保留一个 `_legacy_workflow_state` 派生器，只给本地库兼容持久化使用

这个边界我觉得是干净的：

1. 对前端：
   - `workflow_state` 已经不存在
2. 对本地持久化：
   - 旧数据仍能平滑升级
3. 对未来继续收尾：
   - 真正删掉物理列时，影响面已经被压到最小

这一步比“直接删库字段”更稳，也更符合当前项目仍在快速演进的现实。

---

## 三、前端终于不再靠兼容态猜状态

我今天非常在意的一点是：

`前端不能继续用 workflowState 这种兼容字段做核心准入判断。`

如果继续那样做，后端再怎么重构，前端心智模型还是旧的。

所以前端这轮真正完成了三件事：

1. `coreClient` 类型改成新契约
2. `useWorkspaceStore` 只消费
   - `summary_state`
   - `media_summary`
   - `runtime_state`
   - `capabilities`
   - `active_tasks`
3. 页面准入和事件处理彻底从旧 `workflowState` 迁走

这一步的意义不是“多几个字段”，而是：

`前端终于不再猜，而是直接读事实。`

这会直接降低以后继续做：

1. `planning_only chat`
2. 素材部分就绪
3. `agent/export/media` 并发
4. 错误恢复

这些场景时的分支复杂度。

---

## 四、文档和测试也一起收口了

我不想让这轮重构停在“代码改了，但文档还在骗人”。

所以今天我把两件事一起做了：

1. 重写 `docs/store/01_core_api_ws_contract.md`
2. 更新 `core/README.md`

重点是把下面这些事实写清楚：

1. 公开契约已经不再暴露 `workflow_state`
2. `task.updated` 和 `error.occurred` 不再带它
3. 空项目合法
4. `planning_only` 模式允许需求澄清，但禁用依赖索引的工具
5. `active_tasks` 才是异步过程事实源

同时我也把测试预期收口到了新模型上。

这里有一个小插曲：

`chat` 失败后，项目摘要状态现在应该进入 `attention_required`，而不是继续停在 `editing`。

测试一开始还是旧预期，所以我顺手把它们改掉了。

这个改动很重要，因为它说明：

`summary_state` 表达的是当前项目是否需要用户注意，而不是草案是否仍然存在。`

草案可以继续存在，项目也可以同时处于 `attention_required`。  
这正是这次状态模型拆分之后才能说清楚的事。

---

## 五、我对这轮收尾最满意的地方

我最满意的不是“删掉了一个字段”，而是终于把几种本来就不该混在一起的东西分开了：

1. 项目元数据
2. 素材事实
3. 运行时事实
4. 准入事实
5. 异步任务事实

这意味着后面继续推进时，系统会更像一个真正的本地后端，而不是一个围绕前端兼容字段勉强运行的状态机。

如果以后还要继续做下一轮收尾，我认为顺序应该是：

1. 等确认不再需要旧库兼容后，删掉 `SQLite workflow_state` 物理列
2. 继续减少 `active_task` 这个兼容便利字段的存在感
3. 让更多 UI 派生态直接从 `summary_state + capabilities + active_tasks` 计算

---

## 六、给未来自己的结论

1. 不要再把 `workflow_state` 拉回公开契约
2. 不要用单轴状态替代正交事实
3. 出错后的项目不一定“不可编辑”，但应该进入 `attention_required`
4. 空项目不是异常路径，而是合法起点
5. `capabilities` 比“猜现在像不像 ready”更适合做前端和 `agent` 的准入判断
