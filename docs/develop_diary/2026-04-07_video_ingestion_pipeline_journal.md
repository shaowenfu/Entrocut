# 2026-04-07 视频素材处理全流程打通日志

今天我把 EntroCut 里悬置已久的视频素材处理管线（Video Ingestion Pipeline）真正打通了。

这不仅是把一个“asyncio.sleep”的模拟占位换成了几行视频处理代码，更是一次对“云端与本地如何安全、高效地协同处理重型多媒体数据”的工程实践。

如果说前几轮重构（如 `2026-03-31` 的状态模型收口）把车道铺好了、红绿灯立好了，那么今天我做的事情就是：让第一辆载着真实重型数据的车，在这条车道上跑完全程。

---

## 一、我为什么在这个时候打通素材管线

在之前的架构梳理中，我们已经明确了系统的基本边界：
1. **Core（本地）** 是绝对的权威中心，视频原片绝不上传。
2. **Server（云端）** 是无状态的大模型网关和代理。
3. **Client（前端）** 是状态的忠实消费者。

在 3 月底重构 `store.py` 时，素材导入状态已经按 `pending -> segmenting -> vectorizing -> ready/failed` 这种事实阶段被清晰地定义了出来。但在后端，`_run_assets_import` 这个函数里执行的仍然只是 `await asyncio.sleep(2)`。

这会导致一个很严重的问题：**如果真实的 CPU 密集型任务（视频解码、切分）和 I/O 密集型任务（云端向量化请求）加进来，之前设计的并发策略、WebSocket 实时推送、以及前端骨架屏的感知，能不能真的扛住？** 

如果不尽早用真实负载去压测这条主链，那些精巧的状态机设计就永远只是纸上谈兵。所以，今天我决定把这个 Mock 拔掉。

---

## 二、第一步：我先把重型算力收拢到一个干净的沙盒里

处理视频很容易写得非常脏，特别是在 Python 里。如果直接把 `ffmpeg` 和 `scenedetect` 的调用塞进 `store.py` 里，那个文件很快就会变成一个无法维护的怪兽。

所以我第一步做的是在 `core` 下新建了 `ingestion.py`。
我给这个模块定了一个死规矩：**它是纯无状态的算力函数集合，不碰任何数据库，不发任何 WebSocket 事件，也不管业务逻辑。**

它只暴露三个极其明确的纯函数：
1. `detect_scenes(video_path)` -> 返回 `start_ms`, `end_ms` 列表。
2. `extract_frame_at(...)` -> 返回某一帧的字节流。
3. `extract_and_stitch_frames(...)` -> 把一个 Clip 的 4 个关键帧抽出来，拼成一个带数字水印（1, 2, 3, 4）的 `2x2` 九宫格，直接返回 Base64 字符串。

特别是第三个函数，它完美贯彻了我们的“隐私和降本第一性原理”：大模型不需要看毫无信息量的连续帧，也不需要消耗几百万 token 去看完整视频。它只需要看一张经过精心排版、包含时间暗示的高信息密度图片。而这一切脏活累活（抽帧、排版、写水印、转 Base64），全在本地用户的 CPU 内存里默默完成了。

为了保证这部分核心算法的绝对稳定，我还为 `ingestion.py` 补充了 100% 覆盖率的独立单元测试。

---

## 三、第二步：在 Store 中安全地调度这些“洪水猛兽”

算力函数有了，下一步是把它们塞进 `store.py` 的背景任务 `_run_assets_import` 中。

这里我踩到的第一个坑，也是异步编程里最经典的坑：**如果在 FastAPI 的主 Event Loop 里直接跑 OpenCV 或者 FFmpeg 这种同步阻塞代码，整个后端会瞬间“心跳骤停”，WebSocket 也会断开。**

所以我严格使用了 `await asyncio.to_thread(...)`。
无论是 `detect_scenes` 还是后续批量的 `extract_and_stitch_frames`，全部被扔进了线程池。这样一来，即使本地机器的 CPU 跑满了，`Core` 依然能游刃有余地向前端继续推送 `task.updated` 和 `asset.updated` 进度事件。

在向 Server 请求向量化的环节，我也做了一个务实的折中：**Batching（分批）**。
如果用户导入了一个一小时的视频，切出了几百个 Clip，如果把几百张拼接好的 640x480 的图片 Base64 一次性全塞进 HTTP Request Body 发给 Server，大概率会触发超时或者 Payload Too Large。所以我以 10 个 Clip 为一个 Batch 进行循环请求。这不仅保护了网络，还能让前端的进度条以肉眼可见的平滑幅度向前推进。

---

## 四、第三步：让前端的 Loading 变得真实可信

后端的管线一旦动起来了，前端的展现就变得非常自然。

在 `WorkspacePage.tsx` 里，我没有去重构原有的布局，而是顺着已有的 `processingStage` 字段，做了一个非常符合直觉的骨架覆盖：
- 当一个 Asset 处于 `segmenting` 或 `vectorizing` 时，它的缩略图位置会出现一个优雅的 Spinner，并伴随透明的遮罩和进度百分比。
- 文本会从“镜头切分中...”动态变成“云端特征提取中...”。
- 如果发生异常（比如文件被意外删除、云端服务超时），它会变成一个带 `AlertCircle` 的红色失败卡片。

这不是为了“做 UI 而做 UI”，而是为了让用户能真真切切地感知到：**系统正在为你的素材拼命工作。** 并且，用户完全可以在这个 Loading 的过程中，继续向 Planner 发送 `planning_only` 模式的 Chat 请求（因为上一轮重构已经支持了并发 Slot）。

## 五、测试修复：Mock 的艺术

今天耗时最长的部分其实不是写业务代码，而是修复由于引入真实 I/O 导致的集成测试崩溃。

原来的测试 `test_server_toolchain_integration.py` 习惯了一切都是瞬间发生的。但现在，代码里充满了 `httpx.AsyncClient` 的网络请求和对本地视频文件的强依赖。

我没有选择删除那些测试，也没有放宽断言，而是进行了极其精准的 Mock：
1. 拦截了 `ingestion.detect_scenes` 和 `ingestion.extract_and_stitch_frames`，让它们返回固定的假时间戳和假 Base64。
2. 拦截了 `store.AsyncClient`，模拟 Server 返回了 200 OK。
3. 最有意思的是，在测并发能力（比如“处理素材时仍能进行编辑规划”）的用例里，我故意给 `asyncio.to_thread` 注入了 `await asyncio.sleep(0.5)` 的延迟。这让测试环境完美重现了真实世界的“耗时感”，从而成功断言了各种竞争态下的中间状态机流转。
