# 2026-04-15 毕设论文插图生成与 Word 写入日志

## 今日目标

本次工作的目标不是继续推进产品代码，而是把毕设论文从“离散草稿 + 离散插图资产”收束成一份可继续精修的正式 `docx（Word 文档）` 终稿骨架。

具体目标分为三段：

1. 建立一条稳定的论文插图生成链路。
2. 把 `docs/毕设/workspace/` 下的离散章节草稿灌入学校模板。
3. 在不破坏模板格式的前提下，补齐插图、图题、目录、参考文献与声明页信息。

## 第一阶段：论文插图生成方案切换

一开始并没有直接采用 `Gemini（Google 图像模型）`。

### 1. 废弃的 `HTML + CSS（网页绘图）` 路线

最初曾尝试通过 `HTML + CSS` 生成论文配图，再通过浏览器截图落盘。这个方案的优点是：

1. 图中文字完全可控。
2. 版式可以精确复现。
3. 后续修改成本低。

但它很快暴露出一个根本问题：

`它更像“自己手工画框图”，而不像“高质量科研插图”。`

虽然可以生成结构正确的图，但画面质感偏普通线框图，难以达到用户想要的“顶会 / 顶刊论文配图”观感。因此这一整套 `paper-figures` 方案最终被废弃并清理掉。

### 2. 切换到 `Gemini 3.1 Flash Image Preview`

在放弃网页绘图后，转而采用 `Gemini 3.1 Flash Image Preview` 作为论文插图生成引擎。为此先做了三件基础工作：

1. 在 `docs/毕设/workspace/` 下编写最小可执行脚本 `generate_gemini_figure.py`。
2. 让脚本从同目录 `.env` 中读取 `GOOGLE_API_KEY`。
3. 建立 `gemini_prompts/` 与 `gemini_outputs/` 目录，用于保存专项提示词、完整提示词副本、生成图片与元数据。

这里的关键判断是：

`图像模型的价值不在于替代排版引擎，而在于调动它内化的科研图审美与构图先验。`

也就是说，不应该把它当成“必须严格按几何约束执行的 SVG 生成器”，而应该把它当成“可以在高维空间中自发形成科研图气质的视觉模型”。

## 第二阶段：提示词探索与插图风格收敛

### 1. 第一轮错误方向

最初为了压制乱码和保证文字准确，提示词写得过于死板，包含大量几何结构与禁止项约束。这条路线虽然在形式上“更可控”，但结果恰恰变差了：

1. 输出更像普通架构框图。
2. 视觉层级弱。
3. 画面气质保守、模板化。
4. 容易把模型压回低质量 `block diagram（框图）`。

用户指出方向完全反了，这个判断是正确的。

### 2. 提示词方法论调整

随后对提示词思路做了根本性调整。新的原则变成：

1. 只强约束“文字白名单”。
2. 只提供“核心叙事”和“内容语义”。
3. 尽量少规定几何细节，让模型自由构图。
4. 把提示词从“施工图”改成“创作委托书”。

最终收敛出的专项提示词结构大致是：

1. `Figure Type（图表类型）`
2. `Core Story（核心叙事）`
3. `Content Semantics（内容语义）`
4. `Visual Tone（视觉气质）`
5. `Text Whitelist（文字白名单）`

这个结构的重要意义在于：

`我们只控制不变量，不去控制模型最擅长的视觉组织方式。`

### 3. 风格版本探索：`v1` 到 `v6`

第一张图“系统总体架构图”经历了多轮探索：

1. `v1-v3`：逐步从普通框图走向更有层次的科研图。
2. `v4`：第一次在“论文正式插图感”和“画面高级感”之间取得平衡。
3. `v5`：进一步放开护栏后，画面更强，但开始往“答辩首页主视觉 / 海报感”漂移。
4. `v6`：尝试在 `v5` 的高级感基础上收回海报感，但仍未稳定超过 `v4`。

最终决策是：

1. 以 `v4` 作为论文正式插图的风格基线。
2. 允许后续插图在合适情况下小幅向 `v6` 漂移。
3. 不再采用 `v5` 这种更强海报化方向。

这个经验后来被写入 `docs/毕设/workspace/README.md`，作为后续所有论文插图生成的统一风格约束。

### 4. 批量生成剩余 7 张图

在第一张图风格定稿后，又继续生成了剩余 7 张核心论文插图，分别对应：

1. 系统总体架构图
2. `Editing Agent（剪辑智能体）` 执行闭环图
3. `EditDraft（剪辑草案）` 核心数据模型图
4. 素材处理流图
5. 交互执行流图
6. 关键模块协同图
7. 前后端状态同步与关键接口关系图
8. 典型业务流程时序图

所有正式成品统一保存在：

`docs/毕设/workspace/gemini_outputs/`

对应的专项提示词保存在：

`docs/毕设/workspace/gemini_prompts/`

在完成正式出图后，还清理掉了探索阶段的中间产物：

1. 删除 `v1 / v2 / v3 / v5 / v6` 版本图片与元数据。
2. 删除废弃的 `paper-figures/` 目录。
3. 只保留 `v4` 及正式生成链路所需文件。

## 第三阶段：毕设 Word 模板写入策略设计

### 1. 为什么不能直接暴力灌稿

用户要求把 `docs/毕设/workspace/` 下的离散论文初稿，精确填入学校指定模板 `毕业论文-office-word-mcp版.docx`，并明确提出两个高风险约束：

1. 模板是学校指定的，格式绝不能破坏。
2. 模板中有大量批注，必须先读懂批注中的格式要求再写入。

因此，首先做的不是写，而是测试读取能力与结构约束。

### 2. `office-word-mcp` 与 `docx-mcp` 的分工

经过测试后，形成了一个混合策略：

1. `office-word-mcp` 负责批注读取、正文结构回读、段落级精修与验证。
2. `docx-mcp` 负责补充结构信息，如表格、页面与文档摘要。
3. `python-docx + lxml` 负责真正不稳定的大批量连续写入与底层编号修复。

之所以没有纯用 `MCP（模型上下文协议）` 完成全部写入，是因为在上百段连续正文灌入这种任务上，`MCP` 的稳定性不足，而 `python-docx` 更适合大批量、可重复、可校验的块级写入。

因此，最终采用的不是单一路线，而是：

`先用 MCP 摸清模板结构和约束，再用脚本完成大批量写入，再回到 MCP 做局部精修与回读验证。`

### 3. 批注读取与写入清单

在正式写入前，先读取并确认了模板内的 18 条批注，确认内容涵盖：

1. 封面单双面要求。
2. 题名长度与换行要求。
3. 摘要与关键词字号要求。
4. 目录最多到三级标题。
5. 各级标题排式与空行要求。
6. 盲审版与正式版签名页处理要求。

随后单独生成了一份写入清单：

`docs/毕设/workspace/08_毕业论文写入清单.md`

这份文档把 `00` 到 `07` 的离散草稿拆成了可写入块，并标注了：

1. 应写入的模板区域。
2. 对应锚点文本。
3. 应继承的样式。
4. 受哪些批注约束。
5. 推荐使用的工具与批次顺序。

这一步非常关键，因为它把“直接改终稿”的风险，转化成了“按清单分批执行”的可控任务。

## 第四阶段：论文正文写入与全局问题修复

### 1. 大批量正文写入

在写入清单基础上，创建了脚本：

`docs/毕设/workspace/apply_thesis_to_docx.py`

这个脚本的职责不是替代 `MCP`，而是负责那一段最难稳定的“大批量连续正文写入”。

它完成了以下工作：

1. 替换封面题名与摘要页内容。
2. 写入中英文题名、摘要、关键词。
3. 把第 `1` 到第 `6` 章正文接管模板中的示例正文区。
4. 清除模板残留说明段落。

同时保留模板批注与整体结构不变。

### 2. 第一个致命问题：双重标题编号

写入完成后，用户发现了一个贯穿全局的致命错误：

`标题前同时存在手写编号和 Word 自动编号。`

问题根因在于：

1. 模板的 `一级标题 / 二级标题 / 三级标题` 样式本身就绑定了自动编号。
2. 写入脚本又把源文稿里的显式编号文本一并写进了标题，例如 `1 绪论`、`1.1 研究背景与意义`。

于是文档中同时出现两套编号。

修复方法是：

1. 从所有标题文本中剥离显式编号，只保留标题内容本身。
2. 删除因写入过程残留的空白标题段落。
3. 保留模板样式自带的编号机制。

修复后，标题文本中显式编号数量从 `110` 降到 `0`。

### 3. 第二个全局问题：多级编号层级绑定错误

在去掉显式编号后，用户又发现新的全局错误：

`所有二级标题和三级标题都从 1.xx.xx 开始编号。`

这次问题不在正文内容，而在模板底层的 `numbering.xml` 绑定关系：

1. `一级标题` 绑定 `numId=1, ilvl=0`
2. `二级标题` 错绑到独立 `numId=2, ilvl=0`
3. `三级标题` 错绑到独立 `numId=3, ilvl=0`

这导致二级、三级标题都各自从 `1` 重新起号。

为此，对 `apply_thesis_to_docx.py` 做了进一步增强，加入 `normalize_multilevel_heading_numbering()`，把三种标题统一绑定到同一套多级编号：

1. `一级标题 -> numId=1, ilvl=0`
2. `二级标题 -> numId=1, ilvl=1`
3. `三级标题 -> numId=1, ilvl=2`

并重写了编号模板：

1. 一级：`%1`
2. 二级：`%1.%2`
3. 三级：`%1.%2.%3`

修复后，正文编号终于恢复成真正的章节层级编号。

### 4. 封面与 `AI 工具使用声明` 填充

在正文和编号机制稳定后，又继续补齐：

1. 学号。
2. 导师姓名。
3. `AI 工具使用声明` 两段正式说明文字。
4. `AI` 使用情况两张表。

这里特别遵守了一条原则：

`不臆造无法确认的真实信息。`

因此像学生姓名、日期这类后续才给出的真实字段，在拿到明确值之前都保持占位，不做猜测式填写。

`把图放在对应概念第一次被完整解释的段落后面。`

这样处理的好处是：

1. 阅读流更自然。
2. 图片真正服务论证，而不是装饰章节。
3. 页码与目录后续更容易稳定。

在正文后、声明前新增了 `参考文献` 章节，并暂时填入了 `4` 条符合格式要求的示例占位文献，满足论文当前结构完整性要求。

目录部分没有采用手写假目录，而是插入了真正的 `TOC（目录域）`，并通过修改 `settings.xml` 增加 `updateFields=true`，让文档在打开后自动更新目录。

这一步的核心判断是：

`目录必须是 Word 的真目录，而不是看起来像目录的静态文本。`

否则后续一旦插图、分页或章节字数变化，目录就会立即失真。

## 当前产物与结论

到本轮工作结束时，已经形成了一套完整的论文交付链路：

1. `Gemini` 插图生成链路已经收敛，并固化了风格经验。
2. 学校 `docx` 模板已经成功被主干正文接管。
3. 目录、插图、图题、参考文献、`AI` 工具使用声明都已具备可继续精修的正式骨架。
4. 两个全局性高风险问题，即“显式编号 + 自动编号冲突”和“多级编号绑定错误”，都已定位并修复。

从第一性原理看，这次工作的真正价值不只是“把论文写进 Word”，而是建立了一条以后可以反复复用的流程：

`离散草稿 -> 写入清单 -> 批量灌稿 -> 编号修复 -> 插图落位 -> 目录域更新 -> 最终精修`

这条链路一旦成立，后续剩余工作就不再是“重新从头做一份论文”，而是对现有正式文档继续做局部补完和精修。
