# 系统总体设计初稿

## 3 系统总体设计

### 3.1 设计原则与总体思路

本文所研究的是一个 `Agent（智能体）` 驱动的智能视频剪辑系统，其核心任务并不是复现传统视频编辑软件的全部界面能力，而是围绕用户自然语言意图，完成视频粗剪过程中的候选片段发现、结构组织、局部修订与预览执行。基于这一定位，系统总体设计遵循以下原则。

首先，系统以 `Agent` 驱动作为核心组织方式。对于智能视频剪辑任务而言，用户输入通常是高层语义目标，例如“生成一个 30 秒的高燃滑雪混剪”或“剪一个节奏舒缓的旅行回顾短片”。这类目标无法直接映射为单一步骤的固定命令，而需要系统根据当前上下文持续完成任务分解、候选片段检索、草案编排与局部修订。因此，本文不采用简单的规则拼接流程，也不将系统实现为单次黑箱式生成器，而是以 `Editing Agent（剪辑智能体）` 为中心组织系统能力，使系统能够围绕当前状态持续执行 `select`、`compose`、`evaluate` 和 `revise`。

其次，系统以 `EditDraft（剪辑草案）` 作为统一事实源。传统视频编辑系统往往将状态重心放在前端时间线控件或界面组织逻辑中，而本文认为，面向 `Agent` 的智能剪辑系统必须拥有统一且稳定的结构化编辑表示。只有当候选片段、镜头单元、场景分组和当前剪辑结果被组织进同一事实源时，系统的检索、规划、局部补丁和预览渲染才能保持一致。因此，`EditDraft` 在本文系统中不仅是中间结果，更是贯穿 `Agent` 执行、界面展示和渲染导出的核心状态载体。

再次，系统采用 `local-first（本地优先）` 的总体思路。视频素材通常体量较大，且带有较强隐私属性，若将原始视频大量上传到远端服务器，不仅会显著增加带宽成本，也不利于用户隐私保护。为此，本文将素材导入、镜头切分、关键帧抽样、预览渲染以及项目状态维护等关键能力保留在本地执行，只在必要时通过云端网关调用多模态理解与语义检索相关能力。这样既能充分利用云端模型能力，又能保证系统的交互效率与本地可控性。

最后，系统强调工具边界清晰与模块依赖单向。视频剪辑任务涉及素材处理、语义检索、状态组织、规划控制和渲染执行等多个环节，若这些职责混杂在单一模块中，系统将难以维护和扩展。因此，本文在总体设计上将交互表现、本地中枢与云端能力分离，并将 `retrieve（检索）`、`inspect（判定）`、`patch（修订）`、`preview（预览）` 等能力收束为清晰工具边界，使 `Agent` 负责决策编排，而各模块负责在既定边界内完成相应执行。

基于上述原则，本文将系统总体思路概括为：以自然语言剪辑意图为入口，以 `EditDraft` 为统一状态，以 `client + core + server` 三层架构为支撑，以 `Editing Agent` 为执行中枢，构建一个能够完成素材解析、语义检索、草案生成、局部修订与预览导出的智能视频剪辑 `MVP` 系统。

### 3.2 系统总体架构设计

为支撑 `Agent` 驱动的智能视频剪辑闭环，本文将系统划分为 `client（客户端）`、`core（本地引擎）` 与 `server（云端网关）` 三个层次。三层并不是简单的前后端分层，而是围绕智能剪辑任务的执行路径进行职责划分。

#### 3.2.1 `client` 层设计

`client` 层基于 `Electron + React` 构建，是系统面向用户的交互表现层。该层的主要职责包括：接收用户输入的自然语言剪辑请求；提供素材导入、项目切换和结果浏览界面；展示当前 `EditDraft`、候选片段与预览状态；并通过可视化方式向用户反馈系统正在执行的阶段性动作。

在总体架构中，`client` 层并不承担权威业务事实的持久化任务。其作用更接近于系统的“观察窗”与“控制台”，负责把用户意图传递给本地引擎，并把本地引擎返回的草案状态、执行进度与错误语义呈现出来。通过这种设计，前端界面可以保持相对简洁，同时避免状态逻辑与交互逻辑过度耦合。

#### 3.2.2 `core` 层设计

`core` 层基于 `Python + FastAPI` 构建，是本文系统的本地中枢，也是 `Editing Agent` 的主要执行主体。与 `client` 相比，`core` 直接贴近本地素材与项目状态；与 `server` 相比，`core` 更关注当前项目的具体上下文和本地可执行能力。因此，`core` 在总体架构中承担以下职责。

第一，维护本地项目状态与 `EditDraft`。系统的项目信息、素材索引、当前草案和中间执行状态均应以本地状态为权威来源，从而确保 `Agent` 执行、界面展示与渲染导出围绕同一事实源展开。

第二，执行本地媒体处理任务。包括镜头切分、关键帧抽样、候选片段图像拼接、预览渲染和结果导出等能力都由 `core` 调度本地工具完成。这些能力构成了智能剪辑系统从“语义决策”走向“视频结果”的执行基础。

第三，承担上下文工程与 `Agent` 执行。对于每一次用户请求，`core` 都需要根据当前项目状态、已有草案和局部修改目标裁剪出最有价值的上下文，再由 `planner（规划器）` 决定下一步应调用哪些工具、需要检索什么内容、是否需要更新 `EditDraft`，从而完成 `planner -> tool -> observe -> replanning` 的循环。

#### 3.2.3 `server` 层设计

`server` 层同样基于 `Python + FastAPI` 构建，其定位是系统的云端能力网关，而不是本地项目状态的事实源。该层的主要职责包括：为用户请求提供鉴权与配额控制；代理调用多模态模型或大语言模型接口；对外提供向量化、语义检索、候选理解等服务接口；并统一屏蔽第三方模型密钥与底层服务细节。

在总体架构中，`server` 并不直接处理本地视频文件，也不维护项目级剪辑状态。其作用是为 `core` 提供语义能力扩展，使本地引擎能够在需要时获得多模态理解、检索召回和智能规划支持。通过这一设计，系统实现了“本地执行 + 云端语义能力”的职责分离。

#### 3.2.4 三层协同关系

三层协同的基本过程如下：用户首先在 `client` 中发起项目操作或剪辑请求；`client` 将请求发送至 `core`；`core` 根据当前状态组织上下文并决定下一步动作，必要时调用 `server` 获取语义能力支持；随后 `core` 更新 `EditDraft`、触发本地处理或生成预览结果，并将最新状态推送回 `client` 展示给用户。由此形成一个以 `core` 为中枢、以 `Agent` 为执行逻辑核心、以 `server` 为能力外延的三层协同体系。

### 3.3 `Editing Agent` 执行闭环设计

本文认为，视频粗剪任务的本质是围绕用户意图对候选素材持续执行选择、编排、评估与修订，因此 `Editing Agent` 的执行闭环是系统总体设计中的核心部分。为了让系统具备可持续交互和局部调整能力，本文将 `Agent` 的执行过程设计为如下闭环。

#### 3.3.1 `select`：候选片段选择

`select` 阶段的目标是在原始素材中找到与当前剪辑目标最相关的候选内容。对于视频任务而言，选择不是简单的关键字匹配，而是建立在素材切分、关键帧表示、多模态向量化与语义召回之上的过程。因此，系统首先将视频转化为候选 `clip` 集合，再由 `Agent` 结合当前用户意图与草案缺口生成检索假设，通过语义检索找到最可能满足目标表达的候选片段。

#### 3.3.2 `compose`：草案结构编排

在完成候选片段召回后，系统进入 `compose` 阶段。该阶段的任务不是直接输出最终视频，而是将选中的候选片段组织为结构化的 `EditDraft`。在这一过程中，系统需要确定片段顺序、镜头时长、镜头间关系以及必要的 `scene（场景分组）` 组织方式。通过 `compose`，视频内容第一次从“候选集合”转化为“可执行的剪辑结构”。

#### 3.3.3 `evaluate`：当前结果评估

剪辑并不是一次编排后即可结束的任务，系统还必须判断当前结果是否接近用户目标。因此，在 `evaluate` 阶段，`Agent` 需要围绕当前 `EditDraft`、用户最近请求和预览结果，判断当前草案是否需要进一步调整。该阶段既可由系统内部逻辑基于状态进行判断，也可结合用户对预览结果的反馈完成。换言之，`evaluate` 是连接“草案生成”与“局部修订”的桥梁。

#### 3.3.4 `revise`：局部修订与迭代收敛

若当前结果尚未满足目标，系统进入 `revise` 阶段。与传统“一次性生成”不同，本文强调系统必须具备在已有结构基础上进行局部替换、裁剪、重排和增补的能力。也就是说，系统不能每次都推翻已有结果重新生成，而应尽量围绕当前 `EditDraft` 做最小必要修改，使剪辑过程表现为一种可追踪、可连续、可解释的迭代收敛过程。

#### 3.3.5 `planner -> tool -> observe -> replanning`

为了支撑上述四类核心动作，本文将 `Agent` 内部执行机制抽象为 `planner -> tool -> observe -> replanning`。

其中，`planner` 负责在当前上下文下决定下一步动作；`tool` 负责具体执行，如素材检索、候选判定、草案补丁或预览生成；`observe` 负责读取工具执行结果和最新状态；`replanning` 则表示系统基于新状态继续决定下一步动作。通过这一执行逻辑，`Editing Agent` 具有了面向复杂任务的多步组织能力，也使系统能够围绕 `EditDraft` 实现真正的持续交互。

### 3.4 核心数据模型设计

为保证系统状态一致性与模块协同效率，本文设计了一组围绕剪辑任务组织的核心数据模型，其中心是 `EditDraft`。

#### 3.4.1 `Asset`

`Asset` 表示用户导入的原始视频素材，是系统中所有后续候选片段与剪辑结果的来源。该实体通常包含素材路径、所属项目、时长等基础元信息，用于建立项目与素材之间的映射关系。

#### 3.4.2 `Clip`

`Clip` 表示从原始视频中通过镜头切分获得的候选片段，是系统进行分析与检索的基础单元。每个 `clip` 具有明确的时间边界，并可关联关键帧表示、语义向量和相关元数据。本文强调，`clip` 是“分析/检索单元”，其作用在于支持素材理解与候选召回，而不是直接作为最终渲染结果的最小单位。

#### 3.4.3 `Shot`

`Shot` 是系统中最小可编辑语义单元，也是最终剪辑草案中的关键执行单元。与 `clip` 相比，`shot` 更强调“被选入草案并承担具体表达任务”的角色。一个 `shot` 可以直接引用某个候选 `clip`，也可以在时间范围上做进一步裁剪和局部调整。最终预览与导出通常围绕有序 `shot` 集合展开。

#### 3.4.4 `Scene`

`Scene` 表示对多个 `shot` 的可选工作分组，用于在更高层次上表达片段集合的结构关系。在某些任务中，`scene` 可用于组织“开场”“高潮”“收束”等逻辑段落，从而为 `Agent` 提供更高层级的编排空间。本文将 `scene` 视为“可选结构层”，而不是所有任务都必须存在的固定层级。

#### 3.4.5 `EditDraft`

`EditDraft` 是本文系统的核心数据模型，用于统一承载当前视频剪辑草案。该结构需要能够记录项目级剪辑状态、镜头顺序、时间范围、镜头分组和必要的执行信息，从而使系统在检索、规划、修订和渲染过程中始终围绕同一事实源展开。本文选择 `EditDraft` 作为系统事实源，正是为了避免状态分散在不同模块与不同界面结构中，导致一致性被破坏。

综上，本文的数据模型组织逻辑可以概括为：`Asset` 提供原始素材来源，`Clip` 提供分析与检索基础，`Shot` 承担最小可编辑表达，`Scene` 提供可选的结构化组织，而 `EditDraft` 负责将它们统一收束为可执行的剪辑草案。

### 3.5 核心数据流与状态流转设计

系统总体设计不仅依赖清晰的模块边界，也依赖明确的数据流和状态流转路径。本文将系统中的核心流程划分为两条主线，即素材处理流与交互执行流。

#### 3.5.1 素材处理流

素材处理流从用户导入视频开始。首先，系统在 `core` 层为项目建立本地工作上下文，并记录原始素材路径。随后，系统调用本地媒体处理工具对原始视频进行镜头切分，得到一组具有明确时间范围的候选 `clip`。在此基础上，系统对每个 `clip` 进行多帧抽样，并将关键帧组织为能够表达时序语义的图像表示。之后，系统为候选片段生成多模态语义向量，并建立与素材路径、时间范围等元数据关联的索引。

经过这一流程，连续视频不再只是一个难以直接操作的大文件，而是转化为一个可检索、可比较、可被 `Agent` 操作的候选片段集合。素材处理流的作用正是为后续所有智能剪辑行为建立语义基础。

#### 3.5.2 交互执行流

交互执行流从用户输入自然语言剪辑请求开始。首先，`client` 将用户请求传递给 `core`。随后，`core` 基于当前项目状态、已有 `EditDraft` 和局部修改目标组织上下文，并由 `planner` 决定下一步动作，例如生成检索假设、判定候选片段、更新草案结构或触发预览生成。

当需要外部语义能力时，`core` 向 `server` 发起请求，获取向量化、检索或候选理解结果；当需要本地执行能力时，`core` 则调度素材处理与渲染工具。执行结果被写回 `EditDraft`，系统再将最新状态与必要的运行反馈推送给 `client`。若用户对当前结果提出进一步修改，系统继续围绕新的状态进入下一轮 `planner -> tool -> observe -> replanning`，从而形成持续交互的智能剪辑闭环。

#### 3.5.3 状态一致性设计

为了保证上述流程稳定运行，本文在状态流转上强调三点：第一，`EditDraft` 是所有规划、展示和执行的统一事实源；第二，`client` 只展示和操作来自 `core` 的权威状态，而不独立持有剪辑结果事实源；第三，`server` 只提供能力代理与检索中介，不保存本地项目级权威状态。通过这一设计，系统在交互闭环中能够避免状态分叉、结果冲突和模块间语义不一致问题。

### 3.6 本章小结

本章从系统设计原则、总体架构、`Editing Agent` 执行闭环、核心数据模型以及核心数据流五个方面，对本文所设计的智能视频剪辑系统进行了总体说明。可以看出，本文系统的设计重点并不在于复刻传统视频编辑器的界面形态，而在于围绕自然语言意图、`EditDraft` 事实源和 `Agent` 执行机制，构建一个能够完成素材解析、语义检索、草案生成、局部修订与预览导出的智能视频剪辑闭环。

在此基础上，下一章将进一步展开系统关键模块的设计与实现，重点说明素材解析、多模态表示与语义检索、`Agent Planning` 与工具协同、草案生成与局部修订，以及预览渲染与导出等核心模块的具体实现方式。
