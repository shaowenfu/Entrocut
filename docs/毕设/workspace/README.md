# workspace 工作原则

本目录中的所有文稿，均服务于 `EntroCut` 毕设最终版论文的编写与定稿。

## 总体定位

1. 本工作区完全面向毕设最终版论文，不面向开发日志、阶段复盘或产品路线讨论。
2. 全部表述以“毕设项目最小 `MVP（最小可行产品）` 已具备上线与验收条件”为基准展开。
3. 即使仓库中的工程实现仍可继续迭代，论文写作中也不主动强调“未完成”“仍在收口”“尚未落地”等表述。
4. 若必须描述边界，应写成“当前系统聚焦于 `MVP` 能力边界”或“本文暂不将某能力纳入设计目标”，而不是写成“没有做完”。

## 论文主题约束

1. 本项目的论文主题统一表述为：`Agent（智能体）` 驱动的智能视频剪辑系统。
2. 所有章节都应突出 `Editing Agent（剪辑智能体）` 是系统核心，而不是把项目写成普通的视频编辑器或简单的对话式工具。
3. 叙述系统价值时，应围绕以下主线展开：
   - 自然语言驱动的剪辑意图理解
   - `Agent Planning（智能体规划）`
   - 候选片段检索与理解
   - `EditDraft（剪辑草案）` 驱动的结构化编排
   - 局部修订与预览执行闭环
4. 若提到 `client`、`core`、`server` 架构，其目的应服务于 `Agent` 的执行闭环，而不是仅做常规三层系统介绍。

## 写作口径

1. 统一使用论文语言，不使用产品宣传语言，也不使用开发日记口吻。
2. 全文统一使用“本文”“本系统”，避免混用“本项目”“本课题”“我们”。
3. 首次出现的关键英文术语，采用 `English（中文注释）` 形式。
4. 对系统能力的描述应真实、稳定、可答辩，不夸张，但也不主动弱化。
5. 对边界的描述采用 `Non-goals（非目标）` 口径，例如：
   - 本文聚焦视频粗剪，不涉及复杂特效与多人协作
   - 本文重点研究 `Agent` 驱动的粗剪闭环，不展开精剪工作流
6. 论文中的插图文字默认必须使用中文，只有必要的专业术语才允许保留英文，例如 `Agent`、`Editing Agent`、`EditDraft`、`WebSocket`。

## 内容组织原则

1. 摘要、绪论、系统设计、关键模块实现、实验或案例分析、总结展望等部分，都应围绕同一主问题展开：如何构建一个 `Agent` 驱动的智能视频剪辑系统。
2. 项目中的算法、架构、数据模型、检索与渲染机制，都应作为支撑 `Agent` 执行的视频剪辑能力来写。
3. `EditDraft` 是论文中的核心事实源，应被稳定表述为系统的结构化剪辑表示。
4. `select -> compose -> evaluate -> revise` 应被稳定表述为智能剪辑 `Agent` 的核心工作闭环。

## 质量要求

1. 每一版文稿都应可直接继续演化为最终论文正文，而不是临时草稿式随笔。
2. 每一段落都应回答“它是否服务于最终论文答辩与评审”，若不能，则不写。
3. 所有内容都应优先保证：主题一致、术语统一、逻辑闭合、口径稳定。

## Gemini 配图脚本

当前工作区新增了一套基于 `Gemini` 图像生成模型的论文配图脚本，用于生成科研风格的系统图、流程图和结构图。

### 相关文件

1. 脚本：[generate_gemini_figure.py](/home/sherwen/MyProjects/Entrocut/docs/毕设/workspace/generate_gemini_figure.py)
2. 专项提示词目录：[gemini_prompts](/home/sherwen/MyProjects/Entrocut/docs/毕设/workspace/gemini_prompts)
3. 输出目录：`docs/毕设/workspace/gemini_outputs`
4. 密钥文件：`docs/毕设/workspace/.env`

### 使用方式

先激活当前仓库的 `Python` 环境：

```bash
source /home/sherwen/MyProjects/Entrocut/core/venv/bin/activate
```

然后执行：

```bash
python docs/毕设/workspace/generate_gemini_figure.py \
  --prompt-file docs/毕设/workspace/gemini_prompts/fig-3-1-system-architecture.md
```

如需指定输出文件，可以增加：

```bash
--output docs/毕设/workspace/gemini_outputs/custom_name.png
```

### 设计约定

1. 通用科研图要求写在脚本中的固定提示词里。
2. 每张图只在 `gemini_prompts/` 中编写专项提示词。
3. 专项提示词必须提前规划图片中允许出现的具体文字，并严格限制模型只能渲染这些文字。
4. 每次生成后，脚本会同时保存：
   - 图片文件
   - 完整提示词副本
   - 生成元数据

### 配图风格基线与提示词经验

后续所有论文插图，统一以 `fig-3-1-system-architecture-v4.png` 的整体气质作为正式基线；在合适情况下，可以小幅向 `v6` 漂移，以换取更强的高级感和视觉张力，但不应滑向 `v5` 那种更偏海报和主视觉的表达。

#### 正式基线

1. `v4` 是当前论文正式插图的主参考版本。
2. `v4` 的优点是平衡：既明显优于普通线框图，又没有过强的海报感，适合作为论文正文中的系统图、流程图、结构图风格母版。
3. 后续新图默认先按 `v4` 的控制强度与审美方向生成第一版，再决定是否继续放开。

#### 可接受的风格漂移

1. `v6` 可以作为“轻度增强版”参考方向。
2. 允许适度增强：
   - 色彩层次
   - 区域氛围感
   - 中心焦点
   - 画面精致度
3. 但要控制以下风险：
   - 不要变成答辩封面式主视觉
   - 不要出现明显海报化光效
   - 不要让装饰性强于信息表达
   - 不要让某个次要模块抢走主叙事
4. 若新图开始明显接近 `v5` 的海报感，应回退到 `v4` 的克制度。

#### 从探索过程得到的核心经验

1. 对 `Gemini` 图像模型不能把几何结构规定得过死。越像施工图、线框图、排版说明书，越容易退化成普通框图。
2. 真正需要强约束的只有两件事：
   - 图片中的文字必须全部来自提示词预设
   - 文字与图形必须一次生成完成，不能后覆盖
3. 除了文字白名单，其他内容应尽量改为“高层语义引导”，而不是“底层排版命令”。
4. 提示词不能只写形式要求，还要提供足够的内容语义，让模型知道图真正想表达什么。
5. 如果只告诉模型“画一张系统架构图”，它通常会退回到平庸的箭头加方框；如果告诉它“系统的核心叙事、中心角色、主从关系、信息边界”，它更容易生成论文级信息图。

#### 提示词编写原则

1. 固定提示词负责定义总体审美与生成边界，不负责规定每一条线和每一个框。
2. 专项提示词应优先包含以下五类信息：
   - `Figure Type（图表类型）`
   - `Core Story（核心叙事）`
   - `Content Semantics（内容语义）`
   - `Visual Tone（视觉气质）`
   - `Text Whitelist（文字白名单）`
3. 可以使用 `VISUAL SCHEMA（视觉蓝图）` 的方式给出大致构图方向，例如：
   - `Linear Pipeline（线性数据流）`
   - `Cyclic / Iterative（循环迭代）`
   - `Hierarchical Stack（分层堆叠）`
   - `Parallel / Dual-Stream（双路径并行）`
   - `Central Hub（中心辐射）`
4. 这里的 `VISUAL SCHEMA` 只用于告诉模型“应该朝哪类图形组织方式思考”，不能继续细化成死板的版面说明。
5. 专项提示词中，内容语义必须比形式说明更丰富。要告诉模型：
   - 系统中谁是核心
   - 哪些是支撑角色
   - 哪些关系是主线
   - 哪些边界需要被视觉化
   - 读者第一眼应该看到什么

#### 文字渲染规则

1. 论文插图默认使用中文主导，只有必要术语保留英文。
2. 关键标签优先采用：
   - `中文（English）`
   - `English（中文注释）`
3. 术语必须与论文正文口径一致，例如：
   - `客户端（client）`
   - `本地中枢（core）`
   - `云端网关（server）`
   - `Editing Agent（剪辑智能体）`
   - `EditDraft（剪辑草案）`
4. 标签应短、准、稳，尽量避免长句，以降低乱码和幻觉文字的概率。
5. 若模型在高自由度条件下持续污染额外文字，优先收紧文字部分，不优先收紧构图与审美部分。

#### 审美目标

1. 目标不是生成普通 `block diagram（框图）`，而是生成论文可直接使用的 `research infographic（科研信息图）`。
2. 画面应同时满足：
   - 学术论文可用
   - 答辩展示不显寒酸
   - 信息主次清晰
   - 具有一定高级感和记忆点
3. 图像气质优先追求：
   - 精致
   - 克制
   - 有层次
   - 有焦点
   - 不僵硬
4. 不追求：
   - 炫技式复杂纹理
   - 夸张发光效果
   - 产品海报感
   - 过强 UI 截图感
   - 低端 `PPT` 风模板感

#### 生成流程建议

1. 新图首次出稿时，优先按 `v4` 口径生成。
2. 若结果过于普通，再逐步放开约束，向 `v6` 的高级感小幅漂移。
3. 若结果开始出现明显海报化、主视觉化或文字污染明显增加，应及时回退。
4. 不追求“一次提示词定终局”，而是采用：
   - 先得到稳定可用版
   - 再测试更激进版本
   - 最终在 `v4` 到 `v6` 之间选择最适合作为论文插图的版本
