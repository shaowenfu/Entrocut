# workspace 工作原则

本目录中的所有文稿，均服务于 `EntroCut` 毕设最终版论文的编写与定稿。

## 总体定位

1. 本工作区完全面向毕设最终版论文，不面向开发日志、阶段复盘或产品路线讨论。
2. 全部表述以“毕设项目最小 `MVP（最小可行产品）` 已具备上线与验收条件”为基准展开。
3. 即使仓库中的工程实现仍可继续迭代，论文写作中也不主动强调“未完成”“仍在收口”“尚未落地”等表述。
4. 若必须描述边界，应写成“当前系统聚焦于 `MVP` 能力边界”或“本文暂不将某能力纳入设计目标”，而不是写成“没有做完”。

## 论文主题约束

1. 本项目的论文主题统一表述为：`Agent（智能体）` 驱动的智能视频剪辑系统。
2. 所有章节都应突出 `Editing Agent（剪辑智能体）` 是系统核心，而不是把项目写成普通的视频编辑器或简单的对话式工具。
3. 叙述系统价值时，应围绕以下主线展开：
   - 自然语言驱动的剪辑意图理解
   - `Agent Planning（智能体规划）`
   - 候选片段检索与理解
   - `EditDraft（剪辑草案）` 驱动的结构化编排
   - 局部修订与预览执行闭环
4. 若提到 `client`、`core`、`server` 架构，其目的应服务于 `Agent` 的执行闭环，而不是仅做常规三层系统介绍。

## 写作口径

1. 统一使用论文语言，不使用产品宣传语言，也不使用开发日记口吻。
2. 全文统一使用“本文”“本系统”，避免混用“本项目”“本课题”“我们”。
3. 首次出现的关键英文术语，采用 `English（中文注释）` 形式。
4. 对系统能力的描述应真实、稳定、可答辩，不夸张，但也不主动弱化。
5. 对边界的描述采用 `Non-goals（非目标）` 口径，例如：
   - 本文聚焦视频粗剪，不涉及复杂特效与多人协作
   - 本文重点研究 `Agent` 驱动的粗剪闭环，不展开精剪工作流

## 内容组织原则

1. 摘要、绪论、系统设计、关键模块实现、实验或案例分析、总结展望等部分，都应围绕同一主问题展开：如何构建一个 `Agent` 驱动的智能视频剪辑系统。
2. 项目中的算法、架构、数据模型、检索与渲染机制，都应作为支撑 `Agent` 执行的视频剪辑能力来写。
3. `EditDraft` 是论文中的核心事实源，应被稳定表述为系统的结构化剪辑表示。
4. `select -> compose -> evaluate -> revise` 应被稳定表述为智能剪辑 `Agent` 的核心工作闭环。

## 质量要求

1. 每一版文稿都应可直接继续演化为最终论文正文，而不是临时草稿式随笔。
2. 每一段落都应回答“它是否服务于最终论文答辩与评审”，若不能，则不写。
3. 所有内容都应优先保证：主题一致、术语统一、逻辑闭合、口径稳定。
