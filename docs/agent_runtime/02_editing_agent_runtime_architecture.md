# Editing Agent Runtime Architecture

本文档只定义当前项目 `editing agent（剪辑智能体）` 的方向性运行时架构，不展开具体 schema（结构）、字段、算法或实现细节。

目标只有一个：

`先把智能剪辑 agent 的整体骨架固定下来，后续再逐层补细节。`

---

## 1. 核心判断

当前项目不应该把 `agent` 理解为：

1. 一个会聊天的多模态模型
2. 一个会直接吐最终视频的黑箱生成器
3. 一组零散工具的拼接

更合理的定义是：

`agent = 基于状态工作的规划与执行系统`

它位于：

`用户意图` 和 `可预览 / 可导出的视频结果`

之间，负责持续进行：

1. 对话理解
2. 状态维护
3. 动作选择
4. 工具执行
5. 结果回写

---

## 2. 五层架构

当前方向上，`editing agent` 采用五层架构。

### 2.1 State Layer（状态层）

职责：

1. 维护会话级任务状态
2. 维护当前工作对象
3. 维护执行历史和阶段性结果

这是整个 `agent runtime（智能体运行时）` 的中枢层。

### 2.2 Planner Layer（规划层）

职责：

1. 基于当前状态判断下一步动作
2. 在“回复 / 澄清 / 检索 / patch / 预览”等动作之间做选择

规划层不负责穷举用户话术，也不应该被做成大量 `if-else`。

### 2.3 Tool Layer（工具层）

职责：

1. 提供可执行能力
2. 对外暴露稳定契约
3. 将规划层决策落为实际操作

它是 `agent` 对素材、草案和渲染能力的执行接口。

### 2.4 Memory / Context Layer（记忆 / 上下文层）

职责：

1. 控制每一轮推理看到什么
2. 组织短期上下文、工作上下文和长期偏好
3. 降低信息丢失和上下文污染

这一层服务于规划层，但不等于普通聊天历史。

### 2.5 Execution Loop（执行闭环）

职责：

1. 读取当前状态
2. 选择下一动作
3. 调用工具或更新状态
4. 将结果写回状态
5. 决定是否继续下一轮

这是 `agent` 持续推进任务的最小运行循环。

---

## 3. 五层之间的关系

推荐关系如下：

1. `State Layer`
   - 作为运行时单一事实源
2. `Memory / Context Layer`
   - 从状态中提炼出本轮推理所需上下文
3. `Planner Layer`
   - 基于上下文决定下一动作
4. `Tool Layer`
   - 执行动作
5. `Execution Loop`
   - 负责把整轮流程串起来并更新状态

可以理解为：

`state -> context -> planner -> tools -> state`

---

## 4. 在当前项目中的映射

这五层不是抽象教科书，而是要落到当前项目已有基础上。

### 4.1 World Model（世界模型）作为状态层基础

当前已经基本明确的核心对象有：

1. `Asset`
2. `Clip`
3. `Shot`
4. `Scene`
5. `EditDraft`

这些对象不是工具细节，而是后续 `agent runtime` 的工作对象。

### 4.2 局部编辑上下文是规划层入口之一

当前已经识别出的关键上下文包括：

1. `selected_scene_id`
2. `selected_shot_id`
3. 当前编辑作用域是 `global / scene / shot`

它们会进入后续 `selection context` 设计。

### 4.3 检索与 patch 是执行层关键契约

当前已经识别出的执行层关键对象包括：

1. `retrieval request`
2. `candidate clips`
3. `EditDraftPatch`

它们共同组成“从编辑意图到草案修改”的执行链。

## 5. 当前阶段的设计原则

### 5.1 先定边界，再补能力

优先固定：

1. 状态边界
2. 动作集合
3. 工具契约

而不是先堆复杂 `prompt` 或复杂流程图。

### 5.2 规划层保持薄

规划层应只负责：

1. 判断当前是否已具备执行条件
2. 选择下一步动作类型

不应过度模拟模型思维，也不应写成大量路由分支。

### 5.3 工具层保持少而强

每类工具应：

1. 输入输出结构化
2. 错误语义明确
3. 能被组合使用

### 5.4 上下文工程服务于决策，不服务于堆料

不是把越多信息塞给模型越好，而是让模型在当前步骤只看到：

1. 足够决策的信息
2. 足够调用工具的信息
3. 足够避免重复劳动的信息

### 5.5 执行闭环优先于一次性大推理

系统应优先支持：

1. 小步执行
2. 局部修改
3. 可回看历史动作
4. 可根据结果继续推进

---

## 6. 当前不展开的部分

本文档刻意不展开以下内容，后续逐层设计：

1. `session runtime state` 的具体字段
2. `planner actions` 的具体 schema
3. `retrieval request` 的字段与动态召回逻辑
4. `EditDraftPatch` 的字段与操作集合
5. `selection context` 的字段定义
6. `memory/context` 的裁剪与组织策略
7. 各类工具的 `I/O contract`

---

## 7. 下一步建议顺序

建议后续按下面顺序逐层细化：

1. `State Layer`
   - 先定义 `session runtime state`
2. `Planner Layer`
   - 再定义 `planner actions`
3. `Tool Layer`
   - 再固定 `retrieval / patch / preview` 契约
4. `Memory / Context Layer`
   - 再设计上下文裁剪与组织
5. `Execution Loop`
   - 最后把整个运行闭环串起来

---

## 8. 一句话结论

当前项目接下来不是继续堆局部功能，而是：

`把已经讨论出的 EditDraft、selection、retrieval、patch 等局部成果，收束进一个清晰的五层 agent runtime 架构。`
