# Planner Action Schema

本文档定义当前项目 `editing agent（剪辑智能体）` 的 `planner action schema（规划动作结构）`。

目标不是描述模型怎么思考，而是固定：

1. `Planner Layer（规划层）` 可以输出哪些动作
2. 每类动作的语义边界是什么
3. 动作如何把 `runtimeState -> planner -> tool -> runtimeState` 串成最小闭环

本文档只定义动作层的结构与原则，不展开具体 `Tool I/O contract（工具输入输出契约）` 和 `prompt（提示词）`。

---

## 1. 第一性原理

`chat-to-cut（对话到剪辑）` 要成立，模型不能只给出开放式推理结论，而必须回答一个更具体的问题：

`基于当前运行时状态，下一步系统应该做什么。`

如果没有这一步，系统会退化成两种坏形态：

1. 只会聊天，不会稳定推进任务
2. 只会调工具，但不知道为什么现在要调这个工具

因此，`Planner Layer` 的职责不是穷举用户话术，而是把“下一步决定”压缩成有限、稳定、可执行的动作集合。

一句话定义：

`planner action` 是模型对“下一步系统行为”的结构化决策结果。

---

## 2. 它在整体架构中的位置

当前五层架构里，关系应当是：

1. `State Layer`
   - 提供当前任务事实
2. `Memory / Context Layer`
   - 为本轮规划提炼上下文
3. `Planner Layer`
   - 产出一个 `planner action`
4. `Tool Layer`
   - 执行动作
5. `Execution Loop`
   - 将结果回写到 `runtimeState`

也就是：

`runtimeState -> planner action -> tool execution -> runtimeState`

这条链的本质是：

`state -> decide -> act -> state`

它是 agent 最小可运行闭环。

---

## 3. 设计目标

`planner action schema` 当前只服务 4 件事：

1. 把开放式决策收敛为有限动作集合
2. 把规划层和工具层解耦
3. 让每一步都可记录、可验证、可回写
4. 让系统既保留开放对话，又保留结构化执行

它不负责：

1. 规定模型内部思维步骤
2. 穷举所有用户表达方式
3. 直接承载检索、patch、预览等底层参数细节
4. 替代 `runtimeState`

---

## 4. 为什么必须单独设计动作层

如果没有动作层，`Planner Layer` 和 `Tool Layer` 会直接耦合：

1. 模型一会儿输出解释
2. 一会儿输出半结构化命令
3. 一会儿直接想调工具
4. 系统无法稳定判断这一步到底该执行什么

动作层的必要性在于：

1. 给 `planner` 一个稳定出口
2. 给 `tool` 一个稳定入口
3. 给 `runtimeState.execution` 一个稳定记录对象

因此，动作层不是“多一层包装”，而是：

`规划与执行之间的最小语义边界`

---

## 5. 当前推荐动作集合

当前阶段，建议先固定为 8 类动作。

### 5.1 `reply_only`

作用：

只回复用户，不更新工作事实，不调用工具。

适用场景：

1. 讨论选题
2. 讨论风格
3. 解释当前方案
4. 说明系统能力边界

它回答的是：

`现在只需要交流，不需要推进执行。`

---

### 5.2 `ask_clarification`

作用：

向用户追问关键缺失信息。

适用场景：

1. 目标不清
2. 作用域不清
3. 约束冲突
4. 候选不足且系统无法合理补全

它回答的是：

`当前还不具备执行条件，先补信息。`

---

### 5.3 `update_goal`

作用：

把用户刚刚确认的目标、约束、偏好写入 `Goal State`。

适用场景：

1. 用户明确了时长
2. 用户明确了风格偏好
3. 用户补充了必须出现或禁止出现的内容

它回答的是：

`当前这轮输入主要在更新任务目标。`

---

### 5.4 `set_selection_context`

作用：

明确当前编辑作用域。

适用场景：

1. 用户说“这段”
2. 用户说“第二个镜头”
3. 用户切换了当前 scene / shot

它回答的是：

`当前后续动作应该作用在谁身上。`

---

### 5.5 `create_retrieval_request`

作用：

为当前编辑目标生成素材检索请求。

适用场景：

1. 需要替换镜头
2. 需要补素材
3. 需要找开场、转场、结尾候选

它回答的是：

`下一步应该去找什么素材。`

---

### 5.6 `inspect_candidates`

作用：

对候选素材做比较、筛选、重排或触发更深视觉理解。

适用场景：

1. 候选太多，需要比较
2. 候选相似，需要消歧
3. 仅凭文本摘要无法判断

它回答的是：

`已经找到了候选，下一步要判断谁最适合。`

---

### 5.7 `apply_patch`

作用：

对当前 `EditDraft` 施加一次增量修改。

适用场景：

1. trim 某个 shot
2. replace 某个 shot
3. move / group / ungroup
4. 更新选中对象的局部内容

它回答的是：

`现在可以直接改草案了。`

---

### 5.8 `render_preview`

作用：

请求生成新的预览结果。

适用场景：

1. patch 之后需要给用户看结果
2. 当前草案已经具备预览价值

它回答的是：

`当前需要把草案结果外显给用户。`

---

## 6. 为什么刚好是这 8 类

这 8 类动作不是从用户话术归纳出来的，而是从系统“下一步能做什么”反推出来的。

它们恰好覆盖了 `chat-to-cut` 的最小执行面：

1. 只对话
2. 补信息
3. 更新目标
4. 锁定作用域
5. 找素材
6. 看候选
7. 改草案
8. 看结果

再少，会丢掉关键阶段；再多，会把当前系统过度参数化。

因此，这 8 类动作同时满足：

1. 必要性
2. 当前阶段的充分性

---

## 7. 动作对象应包含什么

每个 `planner action` 至少应包含 3 类信息：

1. `action type`
   - 本次要做什么
2. `why now`
   - 为什么当前状态下要做这个动作
3. `action payload`
   - 这类动作的最小必要参数

也就是说，动作对象不是一句命令，而应是：

`动作类型 + 当前理由 + 最小参数`

这样设计的原因是：

1. 便于执行
2. 便于记录
3. 便于失败后回看
4. 便于后续分析规划质量

---

## 8. 动作进入执行层的前置条件

不是所有动作都可以立即执行。

更合理的做法是：为动作定义最低执行前置条件。

例如：

### 8.1 `ask_clarification`

前置条件：

1. 当前存在关键缺失信息
2. 系统无法用合理默认值继续推进

### 8.2 `create_retrieval_request`

前置条件：

1. 当前目标片段或作用域已基本明确
2. 已具备足够检索线索

### 8.3 `apply_patch`

前置条件：

1. 作用对象明确
2. 当前修改不再依赖额外候选证据

### 8.4 `render_preview`

前置条件：

1. 当前草案已有新结果
2. 预览对后续用户反馈有价值

所以，`Planner Layer` 的核心不是“想清所有步骤”，而是：

`判断当前最适合、且已经具备前置条件的下一动作`

---

## 9. 它如何形成真正闭环

当动作层被引入后，整套系统才能形成真正的运行闭环：

### 9.1 读状态

`runtimeState` 提供：

1. 当前目标
2. 当前草案
3. 当前选区
4. 当前候选
5. 最近动作和失败记录

### 9.2 选动作

`planner` 基于这些事实输出一个 `planner action`

### 9.3 执行动作

`tool layer` 根据动作类型执行：

1. 更新目标
2. 更新选区
3. 生成检索请求
4. 检查候选
5. 应用 patch
6. 生成预览

### 9.4 回写状态

执行结果写回：

1. `Goal State`
2. `Selection State`
3. `Retrieval State`
4. `Draft State`
5. `Execution State`
6. `Conversation State`

然后系统再进入下一轮：

`state -> action -> tool -> state`

这就是“真正闭环”的含义。

没有这一步，系统只有零散能力；有了这一步，系统才是持续推进任务的 agent。

---

## 10. 与其它契约的关系

`planner action schema` 不取代其它契约，而是调度它们。

关系如下：

1. `selection_context`
   - 由 `set_selection_context` 相关动作写入或消费
2. `retrieval_request`
   - 由 `create_retrieval_request` 产出
3. `candidate clips`
   - 由 `inspect_candidates` 消费
4. `EditDraftPatch`
   - 由 `apply_patch` 产出并执行

所以动作层不是业务对象层，而是：

`业务对象之间的运行时调度层`

---

## 11. 当前阶段的非目标

本文档刻意不展开以下内容：

1. 每类动作的字段级 schema
2. 模型输出格式约束
3. 动作失败重试策略
4. 多动作批量执行
5. planner 的 prompt 设计
6. tool 的具体函数签名

这些属于下一阶段细化内容。

---

## 12. 一句话结论

`planner action schema` 的本质，是把模型的“下一步决定”收敛成有限、稳定、可执行的动作对象，并据此把 `runtimeState -> planner -> tool -> runtimeState` 串成真正持续推进剪辑任务的 agent 闭环。
