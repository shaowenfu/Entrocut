# State Layer Design

本文档定义当前项目 `editing agent（剪辑智能体）` 的 `State Layer（状态层）` 设计。

目标不是罗列字段，而是先回答两个问题：

1. 为什么 `chat-to-cut（对话到剪辑）` 必须有状态层
2. 当前阶段最小必要的运行时状态有哪些

本文档只定义方向、边界和分层，不展开具体字段 schema（结构）和持久化实现。

---

## 1. 第一性原理

`chat-to-cut` 的本质不是一次生成，而是一个持续收敛过程：

1. 用户通过对话表达意图
2. 系统形成当前理解
3. 系统对草案做一次局部或全局修改
4. 用户继续反馈
5. 系统基于当前版本继续推进

所以系统不能只依赖：

1. 单轮用户消息
2. 全量聊天记录
3. 数据库里原始项目数据

因为这三者都不能稳定回答以下问题：

1. 当前目标到底是什么
2. 当前草案处于哪一版
3. 当前这句话作用在哪
4. 当前找素材进行到哪
5. 系统刚刚做过什么
6. 哪些对话结论已经变成工作事实

因此必须有 `runtime state（运行时状态）`。

一句话定义：

`State Layer` 是 agent 为了持续完成当前剪辑任务而必须记住的结构化事实板。

---

## 2. 设计目标

`State Layer` 当前只服务 4 件事：

1. 支撑多轮对话下的目标对齐
2. 支撑围绕 `EditDraft` 的持续修改
3. 支撑局部编辑与素材检索
4. 支撑 agent 的动作连续性与可追踪性

它不负责：

1. 取代数据库
2. 保存所有聊天原文
3. 保存所有底层工具中间结果
4. 模拟模型全部思维过程

---

## 3. 推导过程

从 `chat-to-cut` 的最小闭环出发，系统必须稳定记住 6 类事实：

1. 用户到底想做什么
2. 当前剪到什么版本
3. 当前在改哪里
4. 当前找素材进行到哪
5. 系统刚刚做了什么
6. 哪些对话结论已经被吸收

这 6 类事实直接推出当前所需的 6 类状态：

1. `Goal State`
2. `Draft State`
3. `Selection State`
4. `Retrieval State`
5. `Execution State`
6. `Conversation State`

这不是任意分类，而是由任务闭环强制决定的。

---

## 4. 六类状态

### 4.1 Goal State

作用：

保存当前会话已经确认的剪辑目标与约束。

它回答：

1. 这次要做的是什么视频或片段
2. 时长、风格、受众、必须项、禁止项是什么
3. 当前是在做完整视频还是局部片段

没有这一层，系统会把局部反馈误当成全局目标。

---

### 4.2 Draft State

作用：

保存当前工作的剪辑事实版本。

它回答：

1. 当前 `EditDraft` 是什么
2. 当前版本号是什么
3. 当前预览对应哪一版草案
4. 是否存在尚未确认或尚未渲染的修改

没有这一层，系统无法稳定围绕“当前版本”继续协作。

---

### 4.3 Selection State

作用：

保存当前对话正在作用的编辑范围。

它回答：

1. 当前是 `global / scene / shot` 哪一层
2. 当前具体选中了谁
3. 哪些对象或字段被锁定

没有这一层，`这段`、`这里`、`第二个镜头` 这类表达都无法稳定落地。

---

### 4.4 Retrieval State

作用：

保存“找素材”这件事当前进行到哪。

它回答：

1. 最近一次想找什么
2. 当前候选池有哪些内容
3. 哪些候选已做深理解
4. 候选是否足够
5. 上一次为什么没找到

没有这一层，系统会重复检索、丢失失败原因、无法扩召。

---

### 4.5 Execution State

作用：

保存 agent 刚刚执行过的动作与结果。

它回答：

1. 最近做过哪些 `planner action（规划动作）`
2. 已应用了哪些 `EditDraftPatch`
3. 哪些工具调用失败了
4. 当前是否正在执行某个动作

没有这一层，系统无法维持过程连续性，也无法解释自己刚刚改了什么。

---

### 4.6 Conversation State

作用：

保存已经被工作流吸收的对话结论，而不是全量聊天原文。

它回答：

1. 哪些信息已经被用户确认
2. 哪些问题还没解决
3. 最近一次用户反馈是接受、拒绝还是局部不满
4. 当前是否需要继续澄清

没有这一层，系统无法稳定地区分“已确认事实”和“仍待确认的问题”。

---

## 5. 层间关系

这 6 类状态不是平铺堆砌，而是围绕 3 个核心面组织的：

### 5.1 意图面

1. `Goal State`
2. `Conversation State`

解决：

`用户到底想要什么`

### 5.2 草案面

1. `Draft State`
2. `Selection State`

解决：

`当前版本是什么，以及这轮修改作用在哪里`

### 5.3 行动面

1. `Retrieval State`
2. `Execution State`

解决：

`系统为了推进草案，已经做了什么，正在做什么`

因此可以把 `State Layer` 理解为：

`意图面 + 草案面 + 行动面`

---

## 6. 与其它层的关系

`State Layer` 在五层架构中的位置是中枢。

它与其它层的关系如下：

1. `Planner Layer`
   - 读取状态，决定下一步动作
2. `Tool Layer`
   - 执行动作后，把结果写回状态
3. `Memory / Context Layer`
   - 从状态中裁剪出本轮推理所需上下文
4. `Execution Loop`
   - 驱动 `read state -> act -> write state`

所以状态层不是被动存储，而是：

`planner、tool、loop 共享的单一工作事实源`

---

## 7. 设计原则

### 7.1 只保留后续决策真正依赖的事实

状态层不是越大越好。

判断标准只有一个：

`如果不保存这条信息，下一步决策或执行会不会明显变差`

---

### 7.2 区分事实、假设、结果、失败

必须避免把这些混在一起：

1. 用户确认事实
2. agent 当前猜测
3. 工具执行结果
4. 历史失败记录

它们语义不同，后续处理方式也不同。

---

### 7.3 状态必须带版本感

至少要能区分：

1. 当前草案版本
2. 当前预览对应的版本
3. 当前候选是否基于最新目标和最新选区

否则系统会围绕过期结果继续推理。

---

### 7.4 状态服务执行，不服务堆料

不要把状态层做成：

1. 数据库镜像
2. 聊天记录堆栈
3. prompt 垃圾堆

状态层的职责只有一个：

`让 agent 在下一步更准确地决定和执行`

---

## 8. 当前阶段的非目标

本文档明确不讨论以下内容：

1. 各状态的具体字段 schema
2. 状态持久化到数据库的方式
3. 多会话共享状态
4. 长期用户偏好记忆
5. 状态压缩、裁剪和窗口管理
6. 并发协作下的冲突处理

这些属于下一阶段设计，不应现在提前支付复杂度。

---

## 9. 一句话结论

从 `chat-to-cut` 第一性原理推导，当前项目所需的 `State Layer`，本质上就是：

`把目标、当前草案、当前选区、检索过程、执行过程、对话结论，固定成一个可持续推进剪辑任务的运行时事实板。`
