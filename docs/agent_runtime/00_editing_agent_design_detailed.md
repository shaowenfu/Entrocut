# Editing Agent 详细设计

本文档用于系统化回顾前几轮关于 `editing agent（剪辑智能体）` 的讨论，并把已经收敛下来的设计原则、关键问题、工程取舍与建议路径整理成可复盘的书面文档。

本文档是详细版，目标是解释：

1. 我们到底在设计什么
2. 为什么 `EditDraft` 和 `retrieval-first（检索优先）` 路线是必要的
3. 关键工程难点分别是什么
4. 接下来实现时哪些边界不能乱

如果要直接看 `retrieve` 的执行级设计，请继续阅读：

- [Retrieve 执行级设计](./07a_retrieve_execution_design.md)

---

## 1. 核心判断

### 1.1 我们做的不是“会剪视频的聊天机器人”

从第一性原理出发，系统真正要做的是：

`把用户的高层意图，转译成一组可验证、可回滚、可局部修改的剪辑操作，并持续逼近结果。`

这和 `Claude Code / coding agent（代码智能体）` 的本质一致。

两者都是：

1. 理解用户目标
2. 基于结构化世界模型推理
3. 调用工具执行局部修改
4. 根据反馈继续迭代

不同点只是操作对象不同：

1. `coding agent` 操作 `repo / file / symbol / patch`
2. `editing agent` 操作 `asset / clip / shot / scene / edit_draft / render`

所以系统重点不在“生成视频”，而在：

1. 建立稳定的 `world model（世界模型）`
2. 定义最小但足够强的 `editing operations（编辑操作语义）`
3. 让 `agent` 能在结构化状态上做增量决策

### 1.2 为什么必须有结构化中间层

如果没有中间层，只剩两条路：

1. 用户一句话，模型直接产最终视频
2. 用户自己回到传统 `timeline（时间轴）` 工具里操作

这两条路都不符合目标：

1. 第一条是黑箱，不可解释、不可局部编辑、不可验证
2. 第二条会让产品退化成传统剪辑器，只是多了个 AI 插件

因此必须有中间层。

这个中间层不是展示卡片，而是：

`让用户意图、素材理解、局部编辑、工具执行和导出结果能够对齐的工作状态层。`

当前这层已经被收敛为 `EditDraft schema（剪辑草案结构）`。

## 2. EditDraft 分层结论

当前推荐分层为：

1. `Asset`
   - 原始素材
2. `Clip`
   - 分析/检索阶段得到的候选素材单元
3. `Shot`
   - 当前草案中一次具体的素材使用，是最小可编辑语义单元
4. `Scene`
   - 可选的工作分组层，用于局部意图表达和批量操作
5. `EditDraft`
   - 当前草案的容器

核心结论：

1. `clip` 不是最小可编辑语义单元
2. `shot` 才是最小可编辑语义单元
3. `scene` 是可选层，不是必选层
4. `render` 最终应以 `shots` 为执行权威输入

### 2.1 为什么 `shot` 才是最小可编辑语义单元

因为用户真正修改的不是素材本体，而是“这次怎么用素材”。

用户会提出的要求通常是：

1. “第二个镜头太长了”
2. “把这一段换成更近的特写”
3. “把这个镜头挪到后面”
4. “这个片段从这里开始，不要前面那半秒”

这些都不是在修改 `clip` 的客观属性，而是在修改某次具体使用：

1. 用哪个 `clip`
2. 取哪个时间范围
3. 放在第几个位置
4. 是否启用

这正是 `shot` 的职责。

### 2.2 `scene` 为什么是可选层

`scene` 不是客观物理对象，而是当前草案里的工作分组。

只有在以下情况，`scene` 才有必要存在：

1. 用户需要按“段落”修改，而不是逐镜头修改
2. 系统需要对一组连续 `shot` 作为整体做替换、重排、删减
3. 当前草案足够复杂，需要更高层的局部操作目标

如果当前任务只是：

1. 做一个短片段
2. 用户逐镜头指定
3. 草案很短

那么完全可以只有：

`EditDraft -> shots`

不必强行升到：

`EditDraft -> scenes -> shots`

## 3. Storyboard 的角色重新定义

讨论后达成的结论是：

`Storyboard` 不是最终执行层，也不是传统时间轴的替代品。

它更适合作为：

`EditDraft.scenes 的展示视图和交互入口。`

因此：

1. `Storyboard` 可以保留作为前端 UI 组织层
2. 但它不应成为事实源
3. 事实源应该是 `EditDraft`
4. `Storyboard` 只应是 `EditDraft` 的派生表示

这也是前一阶段 contract 对齐时，为什么已经把事实源从 `storyboard` 切换到 `edit_draft`。

## 4. Agent 总体分层

当前推荐的 `editing agent` 总体分层如下。

### 4.1 Perception Layer（感知层）

职责：

1. 从原始 `asset` 中抽取低成本元信息
2. 生成可检索的粗粒度索引对象
3. 在必要时对候选片段做更深理解

它回答的问题是：

`素材里可能有什么可用内容？`

### 4.2 Draft Layer（草案层）

职责：

1. 表达当前剪辑方案
2. 管理 `clip / shot / scene / edit_draft`
3. 作为局部 patch 的作用面

它回答的问题是：

`当前这版视频准备怎么剪？`

### 4.3 Planner Layer（规划层）

职责：

1. 理解用户输入
2. 结合当前 `EditDraft` 形成编辑假设
3. 产出检索请求、编辑计划和草案补丁

它回答的问题是：

`为了更接近用户目标，下一步应该做哪些动作？`

### 4.4 Tool Layer（工具层）

职责：

1. 执行检索
2. 执行裁剪、插入、替换、重排
3. 生成预览和导出结果

它回答的问题是：

`这些计划怎么被落地执行？`

### 4.5 Validation Layer（校验层）

这里刻意不叫 `critic（审查器）`，因为它不负责审美判断。

职责只包括：

1. 检查硬约束是否满足
2. 检查 patch 是否命中目标对象
3. 检查结构是否有效
4. 检查是否存在明显的草案错误

它回答的问题是：

`这次操作在结构和约束上成立吗？`

用户反馈负责回答“喜不喜欢”，校验层负责回答“对不对”。

## 5. 为什么必须走 retrieval-first 路线

### 5.1 全量深理解不可扩展

如果用户上传十几个小时素材，切成上千个 `clip`，不可能对每个片段都做昂贵的多模态精理解。

这既不经济，也没有必要。

真正合理的路径是：

`先形成编辑假设，再围绕假设去找素材证据，只对小批候选做深理解。`

这和人类剪辑师的工作方式一致。

### 5.2 正确路线不是“先看懂全部”，而是“先缩小问题”

系统应该这样工作：

1. 根据用户目标和当前草案，先形成当前段落需要什么素材的假设
2. 用低成本索引宽召回一批候选
3. 对候选做重排与补充理解
4. 选择少量素材进入 `EditDraft`
5. 再根据预览和用户反馈继续修改

所以重点不是“视频理解能力最大化”，而是：

`用最低理解成本，持续找到对当前编辑目标最有用的少量素材。`

## 6. phase 1 检索设计：纯 embedding 主召回

### 6.1 为什么先收窄到单通道召回

当前最重要的不是“把所有信号都用上”，而是先保证主召回语义空间稳定。

如果一开始把：

1. `ASR/OCR`
2. `tags`
3. `shot stats`
4. 其它元数据

都混进默认召回主链，很容易出现两个问题：

1. 视觉整体语义被文本或规则噪声污染
2. 召回失效时无法定位到底是哪种信号造成偏差

因此 phase 1 的明确结论是：

`retrieve` 只使用候选 `clip` 的多模态融合 `embedding` 做主召回。

### 6.2 最小索引对象

当前阶段不做复杂多层索引。

只要求每个候选 `clip` 至少具备：

1. `clip_id`
2. `asset_id`
3. `source_start_ms / source_end_ms`
4. 多帧拼接图生成的多模态融合 `embedding`
5. 可用于后续 `inspect` 的关键帧或缩略图引用

这已经足以支撑：

1. 自然语言到候选 `clip` 的语义召回
2. 候选进入 `inspect`
3. 最终进入 `EditDraft`

### 6.3 深理解不属于 `retrieve`

候选级深理解仍然必要，但它属于 `inspect`，不属于 `retrieve`。

也就是说：

1. `retrieve` 负责找到“可能有用”的候选
2. `inspect` 负责用更贵的视觉判断去确认“哪个最合适”

这样分工更符合人类剪辑师的工作方式，也更节省成本。

---

## 7. 检索策略：不是规则库，而是检索假设

我们已经收敛出一个关键判断：

不要试图预先定义一套覆盖所有视频类型的“检索规则库”。

正确做法是：

`让 agent 根据当前任务生成 retrieval hypothesis（检索假设）。`

例如用户要剪旅游视频开头，系统不应只死盯“床”。

更合理的假设空间可能是：

1. `departure / leaving / transit（出发感）`
2. `morning / wake-up / packing（开始状态）`
3. `arrival / first reveal（抵达揭示）`

每个假设先只映射成一个自然语言 `query`，交给主召回向量空间。

当前阶段不要急着把它拆成多路信号。

所以系统真正要定义的不是“所有规则”，而是：

`retrieval hypothesis -> retrieval request` 的转换机制。

## 8. 动态召回策略

核心结论：

`phase 1` 不追求复杂动态召回，而追求稳定可归因的单次执行。

当前最小实现只做 3 步。

### 8.1 Single-query semantic recall（单查询语义召回）

`planner` 为当前编排缺口生成一个主 `retrieval hypothesis`，再落成一个主 `query`。

`retrieve` 只对这个主 `query` 做一次纯多模态 `embedding` 召回。

目标：

`先把单次主召回跑稳，而不是过早把复杂度塞进自动扩召。`

### 8.2 Candidate normalization（候选归一化）

召回后只做两类轻处理：

1. 结果归一化成统一候选结构
2. 对同一 `asset` 中时间高度重叠的候选做最小近重复抑制

然后直接把候选池交给 `inspect`。

### 8.3 Sufficiency check（充分性检查）

当前阶段只回答一个问题：

`这些候选是否足够进入 inspect。`

判断维度只保留：

1. 候选数量是否过少
2. 最高相似度是否过低

若不足，只返回“不足”状态，不在默认链路自动扩召。

自动扩召是后续能力，不是当前执行主路径。

---

## 9. 多路检索还是纯向量检索

当前结论是：

`phase 1 先坚定使用纯多模态 embedding 主召回。`

原因：

1. 这条路径最接近“看片段整体感觉”的人类初筛方式
2. 系统边界更清楚：`retrieve` 只负责高召回，`inspect` 负责精判
3. 如果当前阶段就引入 `ASR/OCR / metadata` 主融合，复杂度会超过收益
4. 未来若要增加辅助通道，也可以作为增量能力接入，而不需要推翻当前契约

这不意味着纯向量召回没有局限，而是意味着：

1. 当前先接受它的边界
2. 把复杂度留给 `inspect`
3. 把辅助通道视为未来增量能力，而不是当前必选项

## 10. 候选级深理解怎么做

这里不应只选一种方案，而应分层协作。

### 10.1 Offline reusable description（离线可复用描述）

对进入候选池的片段，生成可缓存的文本描述：

1. 发生了什么
2. 主体是谁/是什么
3. 有什么关键对象、动作、场景
4. 画面大致风格如何

优点：

1. 可复用
2. 成本可摊销
3. 适合文本 `planner`

缺点：

1. 信息损失
2. 不一定覆盖所有后续问题

### 10.2 Question-driven multimodal query（问题驱动的多模态查询）

在 planner 不确定时，针对具体问题临时询问多模态模型，例如：

1. “这两个候选哪个更像出发前准备？”
2. “这个镜头里是否真的有人在整理行李？”
3. “这个片段的主体是床边起身，还是纯室内空镜？”

优点：

1. 精度高
2. 面向当前任务

缺点：

1. 成本高
2. 不可完全复用

### 10.3 Multimodal planner（多模态规划器）

直接让基座多模态模型看候选关键帧并决策。

优点：

1. 简单直接
2. 对视觉细节判断能力强

缺点：

1. 成本高
2. 上下文压力大
3. 难以扩展到大量候选
4. 稳定性差

### 10.4 当前建议

主链路优先：

1. `offline reusable description`
2. `question-driven multimodal query`

`multimodal planner` 只在高价值节点使用，不作为默认主流程。

## 11. Agent 跟用户的决策边界

这里的核心问题不是“问多少问题”，而是：

`哪些决策必须由用户给出，哪些可以由 agent 默认补全。`

### 11.1 必须尽量由用户给出的决策

这些属于 `goal-defining decisions（定义目标的决策）`：

1. 视频目的是什么
2. 受众是谁
3. 总时长大致多少
4. 是否有必须出现/禁止出现的内容
5. 风格偏好或不可突破的约束

如果这些不明确，系统很容易方向漂移。

### 11.2 可由 agent 先提案的决策

这些属于 `structural defaults（结构默认值）`：

1. 开头从环境建立还是动作切入
2. 中间按地点推进还是按事件推进
3. 结尾收在人物、物体还是空镜

这些可以让 agent 先出一版，再由用户修正。

### 11.3 默认不逐项询问的决策

这些属于 `micro editing decisions（微观编辑决策）`：

1. clip1 还是 clip2
2. 具体裁掉哪 300ms
3. 两个近似镜头谁更顺

如果这些都逐项问用户，系统会严重失去效率。

### 11.4 总原则

`用户定义目标和硬约束，agent 负责方案生成与微观执行；只有在高不确定或高风险分歧时才升级澄清。`

也就是：

`minimum clarification agent（最小澄清智能体）`

## 12. Selection / Context Engineering 的地位

`selected_scene_id / selected_shot_id` 不属于“剪辑能力本身”，但属于系统能否支持局部编辑的前提。

它们承担的是：

`interaction context（交互上下文）`

也就是告诉 agent：

1. 这句话是在改整个草案
2. 还是在改某个 `scene`
3. 还是在改某个 `shot`

没有这类上下文，下面这些话几乎无法正确执行：

1. “这里太长了”
2. “这一段换掉”
3. “第二个镜头不要”
4. “把这组改得更平静一点”

因此，局部选择状态必须进入事实源或至少进入稳定交互状态，而不能停留在页面临时变量里。

## 13. Clip 需要哪些属性

`clip` 不应试图承载所有理解结果，但必须承载足够支持检索、编排和解释的最小信息。

当前建议至少包含这些类目。

### 13.1 基础定位信息

1. `id`
2. `asset_id`
3. `source_start_ms`
4. `source_end_ms`

### 13.2 检索信息

1. `text_summary`
2. `visual_tags`
3. `asr_text`
4. `ocr_text`
5. `embedding_refs`

### 13.3 编排辅助信息

1. 可选画面特征，如 `shot_scale / camera_motion / subject_count`
2. `eventfulness` 或其他粗可用性信号
3. `quality_signals`

### 13.4 证据信息

1. `keyframe_refs`
2. `thumbnail_ref`

### 13.5 置信信息

1. `metadata_confidence`

原则是：

`clip 只存对下一步决策有帮助的信息，不存所有可能信息。`

## 14. 基座模型是否必须多模态

结论：

1. 基座最好支持 `multimodal capability（多模态能力）`
2. 但系统不应依赖“每一步都让基座直接看视频”

### 14.1 纯文本 planner 的可行性

如果系统已经提供了足够好的结构化文本：

1. clip summary
2. tags
3. ASR/OCR
4. quality signals
5. retrieval scores
6. 相邻上下文

那么纯文本 `LLM` 仍可完成大量工作：

1. 检索规划
2. 候选筛选
3. shot/scene 组织
4. `EditDraftPatch` 生成

### 14.2 什么时候必须临时多模态

当用户要求明显依赖视觉细节判断时：

1. “选表情更自然的”
2. “这个镜头更高级一点”
3. “不要路人挡脸”
4. “哪个更有电影感”

这时纯文本摘要不够，需要临时调用多模态判断。

### 14.3 当前推荐架构

1. `planner` 以文本推理为主
2. `judge / reranker` 按需调用多模态

这样能兼顾成本、扩展性和稳定性。

## 15. Validation 层为什么仍然有必要

这里的关键不是做“第二个会审美的大模型”，而是做最小必要的结构校验。

如果把 `Validation` 理解为：

1. 时长约束检查
2. 空草案检查
3. 重复片段过多检查
4. 是否命中用户锁定字段
5. 是否真的修改了用户选中的对象

那它就是必要的。

如果把它理解成：

1. 再让一个模型审美打分
2. 让系统自嗨式判断“更高级了没有”

那价值就很有限。

因此推荐：

1. 保留 `Validation Layer`
2. 但只做硬约束与结构一致性校验
3. 不把它设计成“自动审美仲裁器”

用户反馈永远是创作层面的最终裁决者。

## 16. Agent 最终应该输出什么

最终合理的输出不是长篇解释文本，也不是直接产成片，而是：

`EditDraftPatch（草案补丁）`

典型 patch 包括：

1. 删除某个 `shot`
2. 裁短某个 `shot`
3. 插入一个新 `shot`
4. 用新候选替换某个 `shot`
5. 重排若干 `shot`
6. 把若干 `shot` 归组为一个 `scene`
7. 锁定某些字段

这和 `coding agent` 输出 `diff（代码补丁）` 是同构关系。

因此，`chat` 的未来形态应是：

1. 形成计划
2. 生成 `EditDraftPatch`
3. 执行 patch
4. 返回新的预览与必要解释

---

## 17. 当前工程优先级建议

如果从投入产出比出发，后续最值得推进的顺序建议如下。

### 第一阶段：定义稳定契约

优先定义：

1. `retrieval request schema`
2. `candidate clip schema`
3. `EditDraftPatch schema`
4. `selection/context schema`

原因：

1. 这是 `planner`、检索系统、前端交互和工具执行的公共边界
2. 这些边界一旦稳定，后续模型和实现可替换

### 第二阶段：做检索优先主链路

先实现：

1. `rough segment` 生成
2. 多路索引
3. 宽召回 + 重排 + 充分性检查
4. 候选级深理解缓存

### 第三阶段：把 chat 真正接到局部编辑上下文

也就是让 `selected_scene_id / selected_shot_id` 真正进入请求上下文。

### 第四阶段：再逐步增强复杂视觉判断

先让系统会稳定工作，再逐步引入更强多模态判断。

## 18. 当前共识总结

本轮讨论已经形成的稳定共识如下。

1. `editing agent` 与 `coding agent` 本质同构
2. `EditDraft` 是必要中间层，不是装饰性 UI 结构
3. `shot` 是最小可编辑语义单元
4. `scene` 是可选工作分组层，不应被强制预设
5. `Storyboard` 更适合作为 `EditDraft` 的展示视图，而不是事实源
6. 系统必须走 `retrieval-first` 路线，不能全量深理解素材
7. 理解必须按需、增量、缓存
8. 检索不是静态规则库，而是 `retrieval hypothesis` 驱动
9. phase 1 的 `retrieve` 应先坚定使用纯多模态 `embedding` 主召回，辅助通道未来再增量接入
10. `planner` 可文本主导，复杂视觉歧义由按需多模态补充
11. `Validation` 只做硬约束和结构校验，不负责审美裁决
12. `EditDraftPatch` 应成为 agent 的标准输出

---

## 19. 明确非目标

为保护复杂度预算，当前阶段明确不做以下事情：

1. 不先追求“全量素材精理解”
2. 不把 `scene` 设计成固定叙事模板
3. 不把 `Storyboard` 当成事实源
4. 不把 `timeline` 作为当前主交互中心
5. 不把多模态大模型直接用作全流程默认 planner
6. 不做虚假的自动审美评分系统
