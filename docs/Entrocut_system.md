# Entrocut架构方案：端云协同版 (Edge-Cloud Synergy)

## 1. 核心设计理念

本架构旨在解决传统“全云端”方案中存在的视频上传带宽瓶颈、存储成本高昂以及用户隐私泄露风险。

我们采用 **“数据不动，算力下沉，特征上云”** 的策略：

- **本地优先 (Local-First)**：原始视频文件（Raw Video）永久驻留在用户本地设备（PC/NAS/边缘服务器），绝不上传。
- **特征上云 (Feature Uplink)**：仅提取极少量的关键帧图像特征（Base64/Embedding）传输至云端进行AI计算。
- **结构化理解 (Structured Intelligence)**：利用多模态大模型对候选片段进行深度分析，输出标准化的 JSON 元数据。

## 2. 系统架构全景流程

### 第一阶段：本地预处理与特征提取 (Local Pre-processing)

**目标**：将非结构化的视频流转化为轻量级的图像特征序列。

1. **镜头切分 (Segmentation)**
    - **工具**：本地运行 PySceneDetect。
    - **逻辑**：采用内容感知检测（Content-aware Detection），识别镜头切换的“硬切点”或“淡入淡出”。
    - **输出**：每个镜头的精确入点（Start Time）和出点（End Time）。
2. **多帧抽样 (Multi-frame Extraction)**
    - **策略**：为了保证 10秒 镜头的理解准确性，不上传视频流，而是进行“时序抽帧”。
    - **标准**：建议每镜头抽取 **4-8 帧** 关键图像（例如：第1, 2, 4, 6, 8, 9秒的画面）。
    - **处理**：在本地内存中将图片转码为 Base64 字符串（可在此步骤增加人脸模糊处理以增强隐私）。

### 第二阶段：云端向量化与索引构建 (Cloud Indexing)

**目标**：建立支持自然语言检索的语义索引库，并实现严格的用户数据隔离。

1. **特征向量化 (Embedding)**
    - **模型**：**阿里云 Qwen3-VL-Embedding**。
    - **输入**：步骤一提取的 4-8 张 Base64 图片序列（作为多图输入）。
    - **输出**：一个高维语义向量（如 1024维），该向量融合了镜头内的时序动作和物体信息。
2. **隔离入库 (Isolated Storage)**
    - **数据库**：**阿里云 DashVector**。
    - **存储结构**：
        - **Vector**：特征向量数据。
        - **Metadata (元数据)**：
            - `user_id`: **关键字段**，用于逻辑隔离不同用户的数据。
            - `file_path`: 视频在用户本地的绝对路径（如 `D:/Videos/2026/vlog.mp4`）。
            - `time_range`: 镜头的起止时间戳。
    - **隐私承诺**：云端数据库中仅存储数学向量和本地路径字符串，**不存储任何视频画面**。

### 第三阶段：检索与结构化理解 (Retrieval & Structured Understanding)

**目标**：这是系统的核心大脑，负责从海量数据中“捞出”目标，并“看懂”画面细节。

1. **语义检索 (Semantic Retrieval)**
    - **输入**：用户的自然语言指令（如：“画面中有红色的耐克运动鞋”）。
    - **过滤机制**：在查询时强制附加 `filter="user_id = 'current_user'"` 条件，确保用户只能检索到自己的私有数据。
    - **召回**：DashVector 返回语义最匹配的 Top-N 个候选镜头（包含本地路径信息）。
2. **深度结构化理解 (Deep Understanding)**
    - **触发条件**：仅针对检索召回的 Top-N 个镜头进行二次精细分析。
    - **模型**：**阿里云 Qwen3-VL-Flash**。
    - **输入**：
        - 候选镜头的 n 张关键帧图片（再次从本地读取或使用缓存）。
        - **Schema指令**：要求模型严格按照预定义的 JSON 格式输出。
    - **输出示例 (JSON)**：
        
        ```
        {
        待设计
        }
        
        ```
        
    - **作用**：此 JSON 数据用于后续的 Agent（代理）交互和复杂业务逻辑判断
    - **协作约束**：Schema（结构规范）以 `docs/schemas/analysis.json` 为准，变更流程参考 `docs/coordination/INTERFACES.md` 与 `docs/coordination/CHANGE_LOG.md`

### 第四阶段：本地自动化剪辑 (Local Production)

**目标**：根据云端返回的指令，在本地完成最终视频的合成。

1. **剪辑指令生成**
    - 系统解析第三阶段生成的 JSON，筛选出符合业务逻辑的片段（例如：`visual_analysis.brand_logo_detected == true`）。
    - 调用大模型生成剪辑列表（EDL - Edit Decision List）。
2. **本地渲染**
    - **工具**：**本地 FFmpeg**。
    - **操作**：
        - 读取本地磁盘的原始高清视频。
        - 根据时间戳进行无损剪切（Stream Copy 模式，速度极快）。
        - 合并（Concat）所有片段。
    - **产出**：在用户本地生成最终的 `Output_Highlight.mp4`。

## 3. 数据隐私与安全架构

- **物理隔离**：原始视频文件从未离开用户硬盘，物理上杜绝了视频泄露的可能。
- **逻辑隔离**：DashVector 数据库通过 `user_id` 字段实施严格的行级权限控制。即使数据库被攻击，攻击者得到的也只是一堆数学向量和无效的本地路径（如 `C:/Users/Admin/...`），无法还原视频内容。
- **脱敏处理（可选）**：在第一阶段上传关键帧前，可在本地运行轻量级算法（如 MediaPipe）对人脸或车牌进行高斯模糊处理。

## 4. 资源与开销分析

| 资源维度 | 传统全云端方案 | **端云协同方案 (本方案)** | 优化效果 |
| --- | --- | --- | --- |
| **上行带宽** | 极高 (上传 1GB 视频) | **极低** (仅上传 ~5MB 图片) | 节省 99.5% |
| **云端存储** | 高 (OSS 存视频) | **零** (仅存向量数据) | 成本降至忽略不计 |
| **云端算力** | 高 (视频解码+推理) | **中** (仅图片推理) | 降低约 60% |
| **本地算力** | 低 | **中** (需运行 FFmpeg 切分) | 利用闲置终端算力 |
| **剪辑耗时** | 慢 (需云端转码下载) | **极快** (本地无损合并) | 秒级出片 |

## 5. 总结

该架构通过 **PySceneDetect (本地切分) + Qwen3-VL (云端多帧理解) + DashVector (隔离存储) + FFmpeg (本地合成)** 的组合，完美实现了在保护用户隐私、极度节省带宽成本的前提下，具备“大模型级”的视频语义理解能力。

系统能够输出标准化的结构化文本（JSON），不仅满足了检索需求，也为后续的自动化媒资管理（MAM）和智能生成提供了高质量的数据基础。
