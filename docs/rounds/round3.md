# Round 3 开发计划（稳定性与可部署收口）

## 1. 背景与目标

### 1.1 背景基线（承接 Round 2）

1. `T1 Unified Workflow State Machine（统一流程状态机）` 已冻结并落地。
2. 主链路已打通：`Client -> Core -> Server -> Core -> Client`。
3. 已确认两个核心遗留问题：
   1. 失败路径人工取证存在 `job_id（任务编号）` 混淆，导致“误判为成功”。
   2. `Client（客户端）` 在 `SUCCEEDED（成功）` 后存在视频不可播放问题。
4. 当前开发态输出目录默认 `/tmp/entrocut_jobs`，不利于本地联调可视化。

### 1.2 Round 3 总目标

1. 将 `MVP（最小可用产品）` 从“能跑通”提升到“可稳定回归”。
2. 修复 `Client` 视频播放链路，确保成功任务可直接预览。
3. 固化失败路径调试口径，保证同一 `job_id` 可追踪到终态与错误码。
4. 形成可部署、可回归、可交接的最小工程基线。

## 2. 范围与非目标

### 2.1 In Scope（本轮范围）

1. 三端稳定性修复（`Client/Core/Server`）。
2. 调试与回归流程标准化（命令、日志、记录口径）。
3. 开发态输出目录策略落地（优先项目内可见目录）。
4. 文档与测试补齐，确保团队可重复执行。
5. 全力提高代码质量，代码清晰简洁

### 2.2 Out of Scope（本轮非目标）

1. 接入真实 `DashScope（模型服务）` / `DashVector（向量数据库）`。
2. 多任务并发调度、优先级队列、断点续跑。
3. 大规模 UI 重设计与复杂交互扩展。

## 3. Round 3 任务拆解（具体）

## 3.1 Core 端任务（Owner: core 工程师）

### R3-CORE-001 失败路径终态一致性

目标：失败输入必须稳定落到 `FAILED`，并携带可枚举错误码。

任务：
1. 校验 `VALIDATING_INPUT（输入校验阶段）` 失败路径：不存在文件、空路径、不支持格式。
2. 明确异步语义：`/jobs/start` 返回 `RUNNING` 仅表示任务已受理，不表示终态。
3. 确保 `GET /jobs/{job_id}` 对同一 `job_id` 最终返回 `FAILED + error.code`。
4. 补充单元测试，覆盖“启动成功但终态失败”的标准流程。

验收：
1. 无效路径任务在固定时间窗内进入 `FAILED`。
2. 错误码稳定为 `VAL_VIDEO_NOT_FOUND`（或对应校验码），不出现“无错误成功”。

### R3-CORE-002 开发态输出目录策略

目标：开发态调试产物可直接在项目内查看。

任务：
1. 约定开发态 `CORE_WORK_ROOT（任务输出根目录）` 指向项目内 `entrocut_jobs`。
2. 保留生产态默认 `/tmp/entrocut_jobs`，避免部署行为漂移。
3. 输出路径在状态对象和日志中一致可见。

验收：
1. 开发态每个任务产物落在 `./entrocut_jobs/<job_id>/`。
2. `artifacts.output_video` 与实际文件路径一致。

### R3-CORE-003 回归测试增强

目标：关键逻辑可独立回归。

任务：
1. 增加状态机终态断言测试（成功、失败、取消）。
2. 增加错误语义测试（`validation_error/runtime_error/external_error`）。
3. 增加输出目录策略测试（开发态与默认态）。

验收：
1. `Core` 关键逻辑测试可一键执行并稳定通过。

## 3.2 Client 端任务（Owner: client 工程师）

### R3-CLIENT-001 视频播放链路修复

目标：`SUCCEEDED` 后可稳定播放最终视频。

任务：
1. ✅ 统一本地文件到可播放 URL 的转换策略，使用 HTTP 协议替代自定义 `media://` 协议。
2. ✅ 兼容中文路径、空格路径与跨平台路径分隔符（通过 `encodeURIComponent` 处理）。
3. ✅ 在播放器增加 `error event（错误事件）` 展示，明确不可播放原因。
4. ✅ 成功态展示"路径 + 可播放状态 + 错误详情（若失败）"。

**实现方案**：
- Core 服务器新增 `/videos/{file_path:path}` 端点，通过 HTTP 提供视频文件
- 客户端使用 `http://127.0.0.1:8000/videos/{encoded_path}` 访问视频
- 移除了之前有 WSL 兼容性问题的自定义 `media://` 协议处理

验收：
1. ✅ 样例视频任务成功后，UI 可播放 `final.mp4`。
2. ✅ 中文路径样例可播放，不出现空白播放器无提示。
3. ✅ WSL 和 Windows 环境下均可正常播放视频。

### R3-CLIENT-002 状态展示与取证一致性

目标：杜绝人工调试时跨 `job_id` 混读状态。

任务：
1. UI 明确展示当前轮询 `job_id` 与终态时间。
2. 历史任务查看时，状态、错误码、输出路径绑定同一任务。
3. 失败任务展示固定字段：`error.type + error.code + message + step`。

验收：
1. 人工调试可基于 UI 直接确认“哪一个 `job_id` 失败/成功”。

### R3-CLIENT-003 客户端测试补齐

目标：播放与状态显示关键分支有自动化保障。

任务：
1. 增加播放器 URL 生成与错误展示测试。
2. 增加终态轮询停止条件测试（仅针对当前任务）。
3. 增加失败态渲染测试（错误码与错误类型）。

验收：
1. `client` 测试通过且覆盖新增分支。

## 3.3 Server 端任务（Owner: server 工程师）

### R3-SERVER-001 Mock Contract 收口

目标：`Mock API（占位接口）` 输入输出稳定，错误可追踪。

任务：
1. 固化 `analyze/edl` 必填字段校验与错误码。
2. 保证 `request_id（请求编号）` 透传与日志一致。
3. 保证 `job_id` 在请求日志中可见，便于跨端对齐。

验收：
1. 契约异常输入均返回统一错误结构。
2. 日志可按 `request_id + job_id` 定位。

### R3-SERVER-002 部署与冒烟基线

目标：`ECS（云服务器）` 上的 `Mock Server` 可稳定运行并可验证。

任务：
1. 完整化部署步骤（构建、启动、健康检查、重启、回滚）。
2. 提供最小 `smoke test（冒烟测试）` 命令集合。
3. 明确运行依赖与环境变量清单。

验收：
1. 公网可访问 `GET /health`。
2. `POST /api/v1/mock/analyze` 与 `POST /api/v1/mock/edl` 可用。

## 3.4 三端联调任务（Owner: 全员）

### R3-INT-001 全链路成功/失败双回归

任务：
1. 成功路径：固定样例视频跑通并可播放。
2. 失败路径：无效路径输入，轮询同一 `job_id` 直到终态，记录错误码。
3. 输出统一联调记录：命令、响应、关键日志、结论。

验收：
1. 成功和失败路径均可重复执行，结果稳定。

### R3-INT-002 调试手册同步

任务：
1. 更新人工调试文档，加入“同一 `job_id` 取证”规范。
2. 更新目录清理规范，避免日志/临时文件污染仓库。

验收：
1. 新成员按文档可独立复现成功/失败链路。
