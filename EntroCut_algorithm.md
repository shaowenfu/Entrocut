## 1. MVP 定义：我们到底交付什么？

我们要交付的 MVP 是 **“Chat-to-Cut 闭环”**。

MVP **不包含**：复杂的时间线拖拽、关键帧动画、多用户协作、云端存储（素材全本地）。

**一句话验收标准**：用户拖入 1 小时的素材，输入“剪辑成 30 秒高燃混剪”，系统能在 1 分钟内生成结构化的 `EditDraft` 预览，并允许用户通过对话微调 `shot / scene` 级别的局部结构。

---

### 2. `core/` (Python Algorithm ) - "The Engine"

**MVP 任务**:

1. **Ingestion Pipeline**: **目标**：用户上传的素材处理，将非结构化的视频流转化为轻量级的图像特征序列。
    1. **镜头切分 (Segmentation)**
        - **工具**：本地运行 PySceneDetect。
        - **逻辑**：采用内容感知检测（Content-aware Detection），识别镜头切换的“硬切点”或“淡入淡出”。
        - **输出**：每个候选 `clip` 的精确入点（Start Time）和出点（End Time）。
    2. **多帧抽样 (Multi-frame Extraction)**
        - **策略**：为了保证候选 `clip` 的理解准确性，不上传视频流，而是进行“时序抽帧”。
        - **标准**：根据之前切分的每个候选 `clip`，每个 `clip` 抽取 **4帧** 关键图像(提取第 1 帧（开头）、最后一帧（结尾），中间两帧选取“帧间差异（Frame Difference）”最大的帧。这样拼接出的图包含了内容的起、止和核心动作点)，使用图片拼接工具拼接成一个4宫格图片，并进行在每一帧的角落加上极小的、人工标注的数字（1, 2, 3, 4)位置编码标记（这样做的目的是节省token，使得一张图片包含了整个候选 `clip` 的关键信息和时间信息，如果大模型能力足够强，它能通过这一张图片理解到内容语义)
        - **处理**：在本地内存中将图片转码为 Base64 字符串，通过云端来调用大模型进行向量化。
    3. **特征向量化 (Embedding)**
        - **模型**：**阿里云 Qwen3-VL-Embedding**。
        - **输入**：提取的 4张关键帧拼接的图片 Base64 。
        - **输出**：一个高维语义向量（如 1024维），该向量融合了 `clip` 内的时序动作和物体信息。
    4. **隔离入库 (Isolated Storage)**
        - **数据库**：**阿里云 DashVector**。
        - **存储结构**：
            - **Vector**：特征向量数据。
            - **Metadata (元数据)**：
                - `user_id`: **关键字段**，用于逻辑隔离不同用户的数据。
                - `file_path`: 视频在用户本地的绝对路径（如 `D:/Videos/2026/vlog.mp4`）。
                - `time_range`: 候选 `clip` 的起止时间戳。
                - 其它元数据待定
        - **隐私承诺**：云端数据库中仅存储数学向量和本地路径字符串，**不存储任何视频画面**。
2. **EditDraft Planner**: 负责把自然语言意图和候选 `clip` 收敛成结构化剪辑草案。
    - **底层原则**：
        - `clip` 是分析/检索单元
        - `shot` 是最小可编辑语义单元
        - `scene` 是可选工作分组层
        - `render` 以有序 `shots` 为最终执行输入
        - 把脚手架收缩到基础设施层，把决策自由留给模型；系统负责提供目标、上下文、工具、状态与错误语义，不试图用固定流程“教 AI 怎么剪”
        - `retrieve / inspect / patch / preview` 是供模型自主组合的能力边界，不是预先写死的工作流；何时搜索、何时深看、何时开始编排，应由模型基于当前目标与上下文自主决策
3. **FFmpeg Wrapper**: 提供无头视频处理能力。
    - *接口*: `render_preview(edit_draft_json) -> stream_url`。
4. 调用底层视频、图像、音频处理库包装而成的供智能剪辑agent使用的工具和技能。
5. **API**: 暴露 `localhost:xxxx` 给 Client 调用。
    1. /chat 接口: 接收用户的自然语言 + 客户端的**轻量级元数据**（不是视频文件），上下文工程，跟用户进行交互，理解和对齐用户意图，作为云端大脑进行决策，指挥core中的工具和技能，完成视频粗剪和细微调整。
        1. 使用Websocket通信
        2. Websocket的event字段类型
            1. chat ：普通的跟用户的交互信息。“您想制作一个什么风格的视频” …，直接流式显示在UI上即可
            2. operations：给core的指令，实现云端对本地工具的调用，或更新 `EditDraft` 中的 `shot / scene` 结构。core在成功执行后会返回成功信息。
            3. notification：通知信息。
    2. 云端向量化与索引构建：**目标**：建立支持自然语言检索的语义索引库，并实现严格的用户数据隔离。
        1. **语义检索 (Semantic Retrieval)**
            - **输入**：由 `agent` 基于当前编排缺口生成的 `retrieval hypothesis（检索假设）` 与自然语言 `query`，而不是机械复用用户原话。
            - **phase 1 原则**：只使用候选 `clip` 的多模态融合 `embedding` 做主召回，不引入 `ASR/OCR` 等辅助通道参与排序，避免噪声污染主语义空间。
            - **过滤机制**：在查询时强制附加 `filter="user_id = 'current_user'"` 条件，确保用户只能检索到自己的私有数据。
            - **召回**：DashVector 返回语义最匹配的 Top-N 个候选 `clip`（包含`file_path`和`time_range`）。`retrieve` 只负责高召回初筛，不负责最终选材。
        2. **深度结构化理解 (Deep Understanding)**
            - **触发条件**：仅针对检索召回的 Top-N 个候选 `clip` 进行二次精细分析。
            - **模型**：**阿里云 Qwen3-VL-Flash**。
            - **输入**：
                - 候选 `clip` 的 n 张关键帧图片（再次从本地读取或使用缓存）。
                - **Schema指令**：要求模型严格按照预定义的 JSON 格式输出。
            - **作用**：此 JSON 数据用于后续的 `inspect（候选精判）`、`EditDraft planning（草案规划）` 与 `shot` 级别选择；精确判断放在 `inspect`，而不是压给 `retrieve`。
