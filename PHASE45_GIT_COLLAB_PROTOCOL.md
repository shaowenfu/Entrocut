# Phase 4/5 同一 Git 工作区协作协议

## 1. 适用范围

本协议适用于五名工程师在同一个 `git workspace（git 工作区）` 内协作推进 `EntroCut` 的 Phase 4/5 以及后续替换真实实现阶段。

## 2. 基本原则

1. 每位工程师只修改邮件里明确授权的文件范围。
2. 不要顺手修边界外问题。
3. 不要提交格式化全仓、批量 rename、无关 import 清理。
4. 不要修改根目录四份设计文档，除非主工程师明确批准。
5. 不要改 `WebSocket event schema`、冻结契约字段和主入口路由。

## 3. 开工前检查

每次开始编码前必须执行：

```bash
git status --short
```

如果看到边界外的脏文件：

1. 先停止
2. 记录文件路径
3. 通知主工程师确认是否继续

## 4. 提交前检查

提交前必须执行：

```bash
git diff --stat
git status --short
```

必须确认：

1. 只包含自己边界内的文件
2. 不包含 `.vscode/`、日志、临时文件、数据库文件
3. 不包含他人工程目录的无关变更

## 5. 冲突规避

1. 主工程师保留修改：
   1. `core/server.py`
   2. `server/main.py`
   3. `client/src/store/useLaunchpadStore.ts`
   4. `client/src/store/useWorkspaceStore.ts`
   5. 根目录任务与协议文档
2. 其它工程师不要主动改这些文件。
3. 如果必须改这些文件，先由主工程师开窗口授权。

## 6. 提交规范

每位工程师提交信息格式建议：

```text
工程师标识: 任务主题
```

例如：

```text
工程师A: 接入真实 segmentation 与 frame extraction
```

## 7. 变更提案要求

遇到以下情况必须先提 `RFC（变更提案）`：

1. 想改 API 契约
2. 想新增跨端字段
3. 想改变状态机流转
4. 想改变任务边界
5. 想改主工程师保留文件

## 8. 验证要求

每个任务完成后至少给出：

1. 最小验证命令
2. 成功路径说明
3. 失败路径说明
4. 残留风险

## 9. 禁止事项

1. 禁止改别人的已分配任务目录
2. 禁止用“顺手”名义扩大修改面
3. 禁止跳过本地验证直接提交
4. 禁止在同一工作区内保留未说明的大量脏文件
